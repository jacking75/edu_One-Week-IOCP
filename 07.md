# Chapter 7. IOCP 환경에서의 스레드 동기화
IOCP 게임 서버는 본질적으로 멀티스레드 환경이다. 여러 워커 스레드가 동시에 완료 통지를 받아 처리하기 때문에, 같은 세션의 데이터에 여러 스레드가 동시에 접근할 수 있다. 이 장에서는 멀티스레드 환경에서 발생하는 문제들을 이해하고, 안전하고 효율적인 동기화 전략을 학습한다.

동기화를 제대로 이해하지 못하면 재현하기 어려운 버그가 발생하고, 과도한 동기화는 성능을 크게 저하시킨다. IOCP의 특성을 이해하고 적절한 동기화 기법을 선택하는 것이 고성능 게임 서버를 만드는 핵심이다.
  

## 7.1 IOCP 멀티스레드 환경 이해
IOCP 서버는 일반적으로 CPU 코어 수만큼의 워커 스레드를 생성한다. 각 워커 스레드는 `GetQueuedCompletionStatus` 함수를 호출하여 완료된 I/O 작업을 가져와 처리한다. 이 과정에서 여러 워커 스레드가 동시에 같은 세션의 I/O 완료를 처리할 수 있다.

```
┌─────────────────────────────────────────────────────────┐
│                    IOCP Completion Port                  │
└─────────┬───────────────┬───────────────┬───────────────┘
          │               │               │
    ┌─────▼─────┐   ┌─────▼─────┐   ┌─────▼─────┐
    │ Worker    │   │ Worker    │   │ Worker    │
    │ Thread 1  │   │ Thread 2  │   │ Thread 3  │
    └─────┬─────┘   └─────┬─────┘   └─────┬─────┘
          │               │               │
          └───────────────┼───────────────┘
                          │
                    ┌─────▼─────┐
                    │  Session  │
                    │   Data    │
                    └───────────┘
```

예를 들어 다음과 같은 상황을 생각해보자. 클라이언트가 동시에 여러 패킷을 전송하고, 서버가 이를 수신한다. WSARecv 완료 통지가 거의 동시에 여러 개 발생할 수 있고, 서로 다른 워커 스레드가 이를 처리한다.

```cpp
// 워커 스레드 1에서 실행
void WorkerThread1()
{
    while (running_)
    {
        DWORD bytesTransferred = 0;
        ULONG_PTR completionKey = 0;
        OVERLAPPED* overlapped = nullptr;
        
        GetQueuedCompletionStatus(iocpHandle_, &bytesTransferred, 
                                 &completionKey, &overlapped, INFINITE);
        
        // Session A의 수신 완료
        Session* session = reinterpret_cast<Session*>(completionKey);
        session->OnRecv(bytesTransferred);  // ← 동시 접근 가능!
    }
}

// 워커 스레드 2에서도 동시에 실행 가능
void WorkerThread2()
{
    // 동일한 코드, 동일한 Session에 접근 가능
}
```

이런 환경에서는 다음과 같은 문제가 발생할 수 있다:

**문제 1: 데이터 경쟁(Data Race)**  
여러 스레드가 동시에 같은 변수를 읽고 쓰면 예측할 수 없는 결과가 발생한다.

```cpp
// 잘못된 예: 동기화 없는 카운터 증가
void Session::OnRecv(DWORD bytesTransferred)
{
    receivedBytes_ += bytesTransferred;  // ← 경쟁 조건!
    // receivedBytes_의 최종 값이 정확하지 않을 수 있다
}
```

**문제 2: 불완전한 상태 노출**  
한 스레드가 여러 변수를 업데이트하는 중에 다른 스레드가 읽으면 일관성이 깨진다.

```cpp
// 잘못된 예: 여러 변수의 원자성 없는 업데이트
void Session::UpdatePosition(int32_t x, int32_t y)
{
    posX_ = x;  // ← 여기서 다른 스레드가 읽으면?
    posY_ = y;  // posX_는 새 값, posY_는 옛 값
}
```

**문제 3: 메모리 순서 문제**  
컴파일러 최적화와 CPU의 명령어 재배치로 인해 코드가 작성된 순서대로 실행되지 않을 수 있다.

```cpp
// 잘못된 예
bool dataReady_ = false;
int data_ = 0;

// Thread 1
void Prepare()
{
    data_ = 42;         // ← 이 두 줄의 순서가
    dataReady_ = true;  // ← 바뀔 수 있다!
}

// Thread 2
void Process()
{
    if (dataReady_)  // true라고 해도
    {
        Use(data_);  // data_가 42가 아닐 수 있다!
    }
}
```

IOCP 환경의 특징을 이해하면 효율적인 동기화 전략을 세울 수 있다. IOCP는 같은 세션에 대한 I/O 완료가 동시에 여러 스레드로 전달될 수 있지만, 각 I/O 작업 자체는 원자적이다. 즉, WSARecv나 WSASend의 완료는 중간 상태 없이 완전히 완료된 상태로만 통지된다.
  

## 7.2 경쟁 조건(Race Condition)과 데드락
경쟁 조건은 두 개 이상의 스레드가 공유 자원에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 상황을 말한다. 게임 서버에서 가장 흔한 경쟁 조건은 세션의 상태 변경과 관련된 것이다.

**전형적인 경쟁 조건 예시**

```cpp
class Session
{
public:
    void Disconnect()
    {
        if (isConnected_)  // ← 체크
        {
            // 여기서 다른 스레드가 끼어들 수 있다!
            isConnected_ = false;  // ← 설정
            closesocket(socket_);
            OnDisconnected();
        }
    }
    
private:
    bool isConnected_ = true;
    SOCKET socket_;
};
```

두 워커 스레드가 동시에 `Disconnect`를 호출하면 다음과 같은 일이 발생한다:

```
시간 순서:
Thread 1: if (isConnected_)  → true
Thread 2: if (isConnected_)  → true  (아직 false로 바뀌지 않음)
Thread 1: isConnected_ = false
Thread 1: closesocket(socket_)
Thread 2: isConnected_ = false
Thread 2: closesocket(socket_)  ← 이미 닫힌 소켓을 또 닫음!
Thread 1: OnDisconnected()
Thread 2: OnDisconnected()  ← 두 번 호출!
```

이 문제는 Check-Then-Act 패턴의 전형적인 경쟁 조건이다. 체크와 실행 사이에 원자성이 보장되지 않는다.

**데드락(Deadlock)**  
데드락은 두 개 이상의 스레드가 서로가 가진 락을 기다리면서 영원히 진행하지 못하는 상황이다.

```cpp
// 잘못된 예: 데드락 발생 가능
class Session
{
public:
    void SendToOther(Session* other, PacketPtr packet)
    {
        std::lock_guard<std::mutex> lock1(mutex_);  // 내 락 획득
        
        // 상대방 세션에 전송
        std::lock_guard<std::mutex> lock2(other->mutex_);  // 상대 락 획득
        other->SendInternal(packet);
    }
};

// 사용 예
// Thread 1: sessionA->SendToOther(sessionB, packet1)
//   → sessionA 락 획득 → sessionB 락 대기
// Thread 2: sessionB->SendToOther(sessionA, packet2)
//   → sessionB 락 획득 → sessionA 락 대기
// → 데드락!
```

이를 시각화하면:

```
Thread 1              Thread 2
   │                     │
   ├─ Lock A             ├─ Lock B
   │   Acquired          │   Acquired
   │                     │
   ├─ Wait for B         ├─ Wait for A
   │   ⏳                │   ⏳
   ▼                     ▼
  BLOCKED              BLOCKED
```

데드락을 방지하는 방법:

1. **락 순서 지정**: 항상 같은 순서로 락을 획득한다.

```cpp
void SendToOther(Session* other, PacketPtr packet)
{
    // 포인터 주소로 순서 결정 (항상 일관된 순서)
    Session* first = this < other ? this : other;
    Session* second = this < other ? other : this;
    
    std::lock_guard<std::mutex> lock1(first->mutex_);
    std::lock_guard<std::mutex> lock2(second->mutex_);
    
    if (first == this)
        other->SendInternal(packet);
    else
        SendInternal(packet);
}
```

2. **std::lock 사용**: 여러 락을 데드락 없이 동시에 획득한다.

```cpp
void SendToOther(Session* other, PacketPtr packet)
{
    std::scoped_lock lock(mutex_, other->mutex_);  // C++17
    // 또는 C++11:
    // std::lock(mutex_, other->mutex_);
    // std::lock_guard<std::mutex> lock1(mutex_, std::adopt_lock);
    // std::lock_guard<std::mutex> lock2(other->mutex_, std::adopt_lock);
    
    other->SendInternal(packet);
}
```

3. **락 계층 구조**: 상위 레벨 락을 먼저 획득한다.

```cpp
// SessionManager (상위) → Session (하위)
// 항상 SessionManager 락을 먼저 획득
```

**라이브락(Livelock)**

데드락과 비슷하지만, 스레드들이 계속 실행은 되지만 진전이 없는 상태다.

```cpp
// 잘못된 예: 라이브락
void TryProcess()
{
    while (!mutex_.try_lock())
    {
        // 락 획득 실패하면 다른 스레드에게 양보
        std::this_thread::yield();
        // 하지만 모든 스레드가 이렇게 하면 계속 양보만 함!
    }
    // 처리
    mutex_.unlock();
}
```
  

## 7.3 Critical Section과 Mutex
Windows는 여러 가지 동기화 기본 요소를 제공한다. 그중 가장 기본적인 것이 Critical Section과 Mutex다.

**Critical Section**

Critical Section은 같은 프로세스 내의 스레드 간 동기화에 사용되는 가볍고 빠른 메커니즘이다.

```cpp
class Session
{
public:
    Session()
    {
        InitializeCriticalSection(&cs_);
    }
    
    ~Session()
    {
        DeleteCriticalSection(&cs_);
    }
    
    void Send(PacketPtr packet)
    {
        EnterCriticalSection(&cs_);
        
        // 임계 영역: 한 번에 한 스레드만 실행
        sendQueue_.push(packet);
        
        LeaveCriticalSection(&cs_);
    }
    
private:
    CRITICAL_SECTION cs_;
    std::queue<PacketPtr> sendQueue_;
};
```

Critical Section의 특징:

- 프로세스 내부에서만 사용 가능하다 (프로세스 간 동기화 불가).
- Mutex보다 빠르다 (커널 모드 전환이 적다).
- 재귀적 락이 가능하다 (같은 스레드가 여러 번 진입 가능).
- 스핀 카운트를 설정할 수 있다 (짧은 대기는 스핀, 긴 대기는 블로킹).

**스핀 카운트 최적화**

```cpp
// 스핀 카운트 설정 (권장: 4000)
InitializeCriticalSectionAndSpinCount(&cs_, 4000);

// 락이 이미 획득된 경우:
// 1. 4000번까지는 busy-wait (스핀)
// 2. 그래도 안 되면 블로킹 (스레드 대기 상태)
```

스핀 카운트를 사용하면 짧은 임계 영역에서 컨텍스트 스위칭 오버헤드를 줄일 수 있다. 멀티코어 시스템에서만 의미가 있다.

**C++ 표준 래퍼**

Win32 API를 직접 사용하는 것보다 C++ 표준 라이브러리를 사용하는 것이 안전하고 편리하다.

```cpp
class Session
{
public:
    void Send(PacketPtr packet)
    {
        std::lock_guard<std::mutex> lock(mutex_);
        sendQueue_.push(packet);
        // lock_guard 소멸자가 자동으로 unlock
    }
    
    bool TrySend(PacketPtr packet)
    {
        std::unique_lock<std::mutex> lock(mutex_, std::try_to_lock);
        if (!lock.owns_lock())
            return false;  // 락 획득 실패
        
        sendQueue_.push(packet);
        return true;
    }
    
private:
    std::mutex mutex_;
    std::queue<PacketPtr> sendQueue_;
};
```

`std::lock_guard`는 RAII 패턴으로 예외 안전성을 보장한다. 예외가 발생해도 소멸자에서 자동으로 락을 해제한다.

**Slim Reader/Writer Lock (SRW Lock)**

읽기가 많고 쓰기가 적은 경우 SRW Lock을 사용하면 성능이 향상된다.

```cpp
class SessionManager
{
public:
    SessionManager()
    {
        InitializeSRWLock(&srwLock_);
    }
    
    SessionPtr GetSession(uint32_t sessionID)
    {
        // 읽기 락 (여러 스레드 동시 가능)
        AcquireSRWLockShared(&srwLock_);
        
        auto it = sessions_.find(sessionID);
        SessionPtr result = (it != sessions_.end()) ? it->second : nullptr;
        
        ReleaseSRWLockShared(&srwLock_);
        return result;
    }
    
    void AddSession(SessionPtr session)
    {
        // 쓰기 락 (배타적)
        AcquireSRWLockExclusive(&srwLock_);
        
        sessions_[session->GetID()] = session;
        
        ReleaseSRWLockExclusive(&srwLock_);
    }
    
private:
    SRWLOCK srwLock_;
    std::unordered_map<uint32_t, SessionPtr> sessions_;
};
```

C++ 표준으로는 `std::shared_mutex`를 사용한다 (C++17):

```cpp
class SessionManager
{
public:
    SessionPtr GetSession(uint32_t sessionID)
    {
        std::shared_lock lock(rwMutex_);  // 읽기 락
        
        auto it = sessions_.find(sessionID);
        return (it != sessions_.end()) ? it->second : nullptr;
    }
    
    void AddSession(SessionPtr session)
    {
        std::unique_lock lock(rwMutex_);  // 쓰기 락
        sessions_[session->GetID()] = session;
    }
    
private:
    std::shared_mutex rwMutex_;
    std::unordered_map<uint32_t, SessionPtr> sessions_;
};
```

락의 성능 비교:

```
빠름 ←─────────────────────────────────────→ 느림
  │                                          │
Interlocked  Critical    SRW      Mutex    Event
Functions    Section    Lock              Semaphore
```
  

## 7.4 InterlockedXXX 함수군 활용
락은 편리하지만 오버헤드가 있다. 간단한 원자적 연산에는 Interlocked 함수군을 사용하는 것이 훨씬 빠르다. 이 함수들은 CPU의 원자적 명령어를 직접 사용하여 락 없이 스레드 안전한 연산을 제공한다.

**기본 원자적 연산**

```cpp
class Session
{
public:
    // 원자적 증가/감소
    void IncrementRefCount()
    {
        InterlockedIncrement(&refCount_);
    }
    
    void DecrementRefCount()
    {
        if (InterlockedDecrement(&refCount_) == 0)
        {
            delete this;
        }
    }
    
    // 원자적 값 설정
    void SetState(SessionState newState)
    {
        InterlockedExchange(reinterpret_cast<LONG*>(&state_), 
                           static_cast<LONG>(newState));
    }
    
    // 원자적 조건부 교환 (Compare-And-Swap)
    bool TryDisconnect()
    {
        SessionState expected = SessionState::CONNECTED;
        LONG result = InterlockedCompareExchange(
            reinterpret_cast<LONG*>(&state_),
            static_cast<LONG>(SessionState::DISCONNECTING),
            static_cast<LONG>(expected)
        );
        
        return result == static_cast<LONG>(expected);
    }
    
private:
    volatile LONG refCount_ = 1;
    volatile SessionState state_ = SessionState::CONNECTED;
};
```

**C++ 표준 atomic 사용**

C++11부터는 `std::atomic`을 사용하는 것이 더 권장된다. 플랫폼 독립적이고 타입 안전하다.

```cpp
class Session
{
public:
    void IncrementRefCount()
    {
        refCount_.fetch_add(1, std::memory_order_relaxed);
    }
    
    void DecrementRefCount()
    {
        if (refCount_.fetch_sub(1, std::memory_order_acq_rel) == 1)
        {
            delete this;
        }
    }
    
    bool TryDisconnect()
    {
        SessionState expected = SessionState::CONNECTED;
        return state_.compare_exchange_strong(
            expected,
            SessionState::DISCONNECTING,
            std::memory_order_acq_rel
        );
    }
    
    SessionState GetState() const
    {
        return state_.load(std::memory_order_acquire);
    }
    
private:
    std::atomic<int32_t> refCount_{1};
    std::atomic<SessionState> state_{SessionState::CONNECTED};
};
```

**메모리 순서(Memory Order)**

`std::atomic`의 강력한 기능은 메모리 순서를 명시할 수 있다는 것이다. 이를 통해 성능과 정확성 사이의 균형을 조절할 수 있다.

- `memory_order_relaxed`: 순서 보장 없음, 가장 빠름
- `memory_order_acquire`: 읽기 배리어 (이후 명령어가 앞으로 이동 안 됨)
- `memory_order_release`: 쓰기 배리어 (이전 명령어가 뒤로 이동 안 됨)
- `memory_order_acq_rel`: acquire + release
- `memory_order_seq_cst`: 순차 일관성, 가장 엄격하고 느림 (기본값)

```cpp
// Producer-Consumer 패턴
class DataProducer
{
public:
    void Produce(int value)
    {
        data_ = value;
        ready_.store(true, std::memory_order_release);
        // release: data_ 쓰기가 ready_ 쓰기보다 먼저 완료됨을 보장
    }
    
private:
    int data_;
    std::atomic<bool> ready_{false};
};

class DataConsumer
{
public:
    int Consume()
    {
        while (!ready_.load(std::memory_order_acquire))
        {
            // acquire: ready_ 읽기 후 data_ 읽기가 재배치 안 됨
        }
        return data_;
    }
    
private:
    int data_;
    std::atomic<bool> ready_{false};
};
```

**ABA 문제**

Compare-And-Swap을 사용할 때 주의해야 할 ABA 문제가 있다.

```cpp
// 잘못된 예: ABA 문제
struct Node { int value; Node* next; };
std::atomic<Node*> head;

void Push(Node* newNode)
{
    Node* oldHead = head.load();
    newNode->next = oldHead;
    
    // 여기서 다른 스레드가:
    // 1. oldHead를 Pop
    // 2. 새 노드를 Push
    // 3. oldHead를 다시 Push (같은 주소!)
    
    while (!head.compare_exchange_weak(oldHead, newNode))
    {
        newNode->next = oldHead;
    }
    // oldHead가 같은 주소여도 내용은 다를 수 있다!
}
```

ABA 문제 해결 방법:

1. **버전 카운터 추가**

```cpp
struct TaggedPointer
{
    Node* ptr;
    uint32_t tag;
};

std::atomic<TaggedPointer> head;

void Push(Node* newNode)
{
    TaggedPointer oldHead = head.load();
    do
    {
        newNode->next = oldHead.ptr;
    } while (!head.compare_exchange_weak(
        oldHead, 
        {newNode, oldHead.tag + 1}
    ));
}
```

2. **Hazard Pointer 사용** (고급 기법)

3. **Epoch-Based Reclamation** (고급 기법)

게임 서버에서는 보통 ABA 문제가 발생할 가능성이 낮은 구조를 선택하거나, Concurrency Runtime의 검증된 컨테이너를 사용하는 것이 안전하다.
  

## 7.5 세션별 작업 직렬화 기법
IOCP 환경에서 같은 세션에 대한 여러 I/O 완료가 동시에 처리되면 복잡한 동기화가 필요하다. 이를 해결하는 강력한 기법이 세션별 작업 직렬화(Per-Session Job Serialization)다. 핵심 아이디어는 **같은 세션의 작업은 한 번에 하나씩만 처리되도록 보장**하는 것이다.

**기본 원리**

각 세션마다 작업 플래그를 두고, 현재 처리 중인지 여부를 원자적으로 체크한다.

```cpp
class Session
{
public:
    void ProcessRecv(DWORD bytesTransferred)
    {
        // 이미 처리 중이면 큐에만 추가
        if (!TryBeginProcess())
        {
            EnqueueJob([this, bytesTransferred]() {
                HandleRecv(bytesTransferred);
            });
            return;
        }
        
        // 직접 처리
        HandleRecv(bytesTransferred);
        
        // 큐에 쌓인 작업들 처리
        ProcessJobQueue();
        
        EndProcess();
    }
    
private:
    bool TryBeginProcess()
    {
        int32_t expected = 0;
        return isProcessing_.compare_exchange_strong(
            expected, 1, std::memory_order_acq_rel
        );
    }
    
    void EndProcess()
    {
        isProcessing_.store(0, std::memory_order_release);
    }
    
    void ProcessJobQueue()
    {
        while (true)
        {
            Job job;
            if (!jobQueue_.try_pop(job))
                break;
            
            job();  // 작업 실행
        }
    }
    
    std::atomic<int32_t> isProcessing_{0};
    concurrency::concurrent_queue<Job> jobQueue_;
};
```

이 기법의 장점:

1. **락 프리**: Critical Section이나 Mutex 없이도 안전하다.
2. **순서 보장**: 같은 세션의 작업이 발생 순서대로 처리된다.
3. **성능**: 경합이 없을 때는 직접 처리, 있을 때만 큐잉한다.

작동 방식을 시각화하면:

```
Thread 1이 Session A 처리 중:

Thread 1          Thread 2          Thread 3
   │                 │                 │
   ├─ Session A      ├─ Session A      ├─ Session B
   │  TryBegin: ✓    │  TryBegin: ✗    │  TryBegin: ✓
   │  → Process      │  → Enqueue       │  → Process
   │                 │                 │
   ├─ Process        │                 ├─ Process
   │  Queued Jobs    │                 │
   │                 │                 │
   ├─ EndProcess     │                 ├─ EndProcess
   │                 │                 │
```

**개선된 구현**

실제로는 더 정교한 구현이 필요하다. 작업 큐에 추가한 후 처리가 끝나버리면 큐의 작업이 고아가 될 수 있다.

```cpp
class Session
{
public:
    void ProcessRecv(DWORD bytesTransferred)
    {
        EnqueueJob([this, bytesTransferred]() {
            HandleRecv(bytesTransferred);
        });
        
        TryProcessJobs();
    }
    
private:
    void TryProcessJobs()
    {
        // 처리 중이 아니면 처리 시작
        if (!TryBeginProcess())
            return;
        
        // 큐가 빌 때까지 처리
        while (true)
        {
            Job job;
            if (!jobQueue_.try_pop(job))
            {
                // 큐가 비었으면 종료
                EndProcess();
                
                // Double-Check: 끝내는 순간 추가된 작업 확인
                if (!jobQueue_.empty() && TryBeginProcess())
                    continue;  // 다시 처리
                
                break;
            }
            
            job();  // 작업 실행
        }
    }
    
    bool TryBeginProcess()
    {
        int32_t expected = 0;
        return isProcessing_.compare_exchange_strong(
            expected, 1, std::memory_order_acq_rel
        );
    }
    
    void EndProcess()
    {
        isProcessing_.store(0, std::memory_order_release);
    }
    
    void EnqueueJob(Job job)
    {
        jobQueue_.push(std::move(job));
    }
    
    std::atomic<int32_t> isProcessing_{0};
    concurrency::concurrent_queue<Job> jobQueue_;
};
```

Double-Check 패턴은 다음 상황을 방지한다:

```
Thread 1                  Thread 2
   │                         │
   ├─ Pop job: ✓             │
   ├─ Execute job            │
   ├─ Pop job: ✗ (empty)     │
   ├─ EndProcess             ├─ EnqueueJob  ← 여기서 추가됨!
   │                         │
   ├─ Check empty again      │
   ├─ TryBegin: ✓            │
   ├─ Pop & Execute          │
   │                         │
```
  

## 7.6 참조 카운팅(Reference Counting)을 통한 안전한 세션 관리
IOCP 환경에서 가장 까다로운 문제 중 하나는 세션의 수명 관리다. 여러 워커 스레드가 동시에 세션을 참조하고 있을 때, 어느 시점에 세션을 안전하게 삭제할 수 있을까?

**문제 상황**

```cpp
// 잘못된 예
void WorkerThread()
{
    Session* session = GetSessionFromCompletionKey();
    
    // 여기서 다른 스레드가 세션을 삭제할 수 있다!
    
    session->OnRecv(bytesTransferred);  // ← 크래시!
}
```

**shared_ptr 사용**

C++의 `std::shared_ptr`은 자동 참조 카운팅을 제공한다.

```cpp
class Session : public std::enable_shared_from_this<Session>
{
public:
    void RegisterRecv()
    {
        // I/O 작업 중에는 참조 카운트 증가
        AddRef();
        
        DWORD flags = 0;
        DWORD bytes = 0;
        
        int result = WSARecv(socket_, &wsaBuf_, 1, &bytes, &flags,
                            &recvOverlapped_, nullptr);
        
        if (result == SOCKET_ERROR && WSAGetLastError() != WSA_IO_PENDING)
        {
            ReleaseRef();  // 실패 시 카운트 감소
        }
    }
    
    void OnRecv(DWORD bytesTransferred)
    {
        // 처리
        HandleRecv(bytesTransferred);
        
        // I/O 완료 - 참조 카운트 감소
        ReleaseRef();
        
        // 다음 수신 등록
        RegisterRecv();
    }
    
private:
    void AddRef()
    {
        auto self = shared_from_this();
        pendingIOCount_.fetch_add(1);
    }
    
    void ReleaseRef()
    {
        if (pendingIOCount_.fetch_sub(1) == 1)
        {
            // 마지막 참조 - 정리 작업
            OnAllIOCompleted();
        }
    }
    
    std::atomic<int32_t> pendingIOCount_{0};
};

using SessionPtr = std::shared_ptr<Session>;
```

**Completion Key에 shared_ptr 저장**

문제는 IOCP의 Completion Key가 `ULONG_PTR`(64비트 포인터 크기)라는 것이다. `shared_ptr`은 제어 블록 포인터까지 포함하므로 단순히 저장할 수 없다.

해결 방법은 OVERLAPPED 구조체를 확장하여 `shared_ptr`을 포함시키는 것이다.

```cpp
struct IOContext
{
    OVERLAPPED overlapped;
    SessionPtr session;  // shared_ptr 저장
    IOType ioType;       // RECV, SEND 등
};

class Session
{
public:
    void RegisterRecv()
    {
        // IOContext 생성 (세션 참조 포함)
        auto ioContext = new IOContext{};
        ZeroMemory(&ioContext->overlapped, sizeof(OVERLAPPED));
        ioContext->session = shared_from_this();  // 참조 카운트 증가
        ioContext->ioType = IOType::RECV;
        
        DWORD flags = 0;
        DWORD bytes = 0;
        
        int result = WSARecv(socket_, &wsaBuf_, 1, &bytes, &flags,
                            &ioContext->overlapped, nullptr);
        
        if (result == SOCKET_ERROR && WSAGetLastError() != WSA_IO_PENDING)
        {
            delete ioContext;  // 실패 시 정리 (shared_ptr 자동 해제)
        }
    }
};

// 워커 스레드
void WorkerThread()
{
    DWORD bytesTransferred = 0;
    ULONG_PTR completionKey = 0;
    OVERLAPPED* overlapped = nullptr;
    
    GetQueuedCompletionStatus(iocpHandle_, &bytesTransferred,
                             &completionKey, &overlapped, INFINITE);
    
    // OVERLAPPED에서 IOContext 복원
    auto ioContext = reinterpret_cast<IOContext*>(overlapped);
    SessionPtr session = ioContext->session;  // shared_ptr 복사 (안전!)
    
    // 처리
    if (ioContext->ioType == IOType::RECV)
    {
        session->OnRecv(bytesTransferred);
    }
    
    // IOContext 삭제 (shared_ptr 자동 해제)
    delete ioContext;
    
    // 이 스코프를 벗어나면 session의 참조 카운트 감소
    // 마지막 참조였다면 Session 자동 삭제
}
```

이 방식의 장점:

1. **자동 메모리 관리**: 수동 AddRef/ReleaseRef 불필요
2. **안전성**: 사용 중인 세션은 절대 삭제되지 않음
3. **단순성**: 복잡한 수명 관리 로직 불필요

**주의사항: 순환 참조**

`shared_ptr`을 사용할 때는 순환 참조를 주의해야 한다.

```cpp
// 잘못된 예: 순환 참조
class Session
{
    SessionPtr partner_;  // 상대 세션 참조
};

// Session A → Session B → Session A
// → 둘 다 참조 카운트가 0이 되지 않아 메모리 누수!
```

해결 방법은 `std::weak_ptr` 사용:

```cpp
class Session
{
public:
    void SetPartner(SessionPtr partner)
    {
        partner_ = partner;  // weak_ptr로 저장
    }
    
    void SendToPartner(PacketPtr packet)
    {
        auto partner = partner_.lock();  // shared_ptr로 변환 시도
        if (partner)
        {
            partner->Send(packet);
        }
    }
    
private:
    std::weak_ptr<Session> partner_;  // 순환 참조 방지
};
```
  

## 7.7 전체 서버 코드
이제 이 챕터에서 학습한 모든 동기화 기법을 통합한 전체 서버 코드를 살펴본다.

**Session.h**

```cpp
#pragma once
#include <memory>
#include <atomic>
#include <functional>
#include <concurrent_queue.h>
#include "RingBuffer.h"
#include "Packet.h"

enum class SessionState : int32_t
{
    CONNECTING = 0,
    CONNECTED = 1,
    DISCONNECTING = 2,
    DISCONNECTED = 3
};

enum class IOType
{
    RECV,
    SEND
};

struct IOContext
{
    OVERLAPPED overlapped;
    std::shared_ptr<class Session> session;
    IOType ioType;
    
    IOContext()
    {
        ZeroMemory(&overlapped, sizeof(OVERLAPPED));
    }
};

class Session : public std::enable_shared_from_this<Session>
{
public:
    using Job = std::function<void()>;
    
    explicit Session(SOCKET socket);
    ~Session();
    
    // 기본 정보
    uint32_t GetID() const { return sessionID_; }
    SOCKET GetSocket() const { return socket_; }
    
    // 상태 관리 (원자적)
    SessionState GetState() const 
    { 
        return state_.load(std::memory_order_acquire); 
    }
    
    bool IsConnected() const
    {
        return GetState() == SessionState::CONNECTED;
    }
    
    bool TryDisconnect()
    {
        SessionState expected = SessionState::CONNECTED;
        return state_.compare_exchange_strong(
            expected,
            SessionState::DISCONNECTING,
            std::memory_order_acq_rel
        );
    }
    
    // I/O 작업
    bool RegisterRecv();
    bool RegisterSend();
    void Send(PacketPtr packet);
    
    // I/O 완료 처리
    void OnRecv(DWORD bytesTransferred);
    void OnSend(DWORD bytesTransferred);
    
    // 인증 관련
    bool IsAuthenticated() const { return isAuthenticated_.load(); }
    void SetAuthenticated(bool value) { isAuthenticated_.store(value); }
    
    const std::string& GetUserID() const { return userID_; }
    void SetUserID(const std::string& userID) { userID_ = userID; }
    
    // 위치 관련 (원자적)
    void SetPosition(int32_t x, int32_t y)
    {
        posX_.store(x, std::memory_order_relaxed);
        posY_.store(y, std::memory_order_relaxed);
    }
    
    void GetPosition(int32_t& x, int32_t& y) const
    {
        x = posX_.load(std::memory_order_relaxed);
        y = posY_.load(std::memory_order_relaxed);
    }
    
private:
    // 작업 직렬화
    void EnqueueJob(Job job);
    void TryProcessJobs();
    bool TryBeginProcess();
    void EndProcess();
    
    // 패킷 처리
    void ProcessPackets();
    std::vector<PacketPtr> ExtractPackets();
    
    // 연결 종료
    void Disconnect();
    void OnDisconnected();
    
private:
    // 기본 정보
    static std::atomic<uint32_t> s_nextSessionID;
    uint32_t sessionID_;
    SOCKET socket_;
    
    // 상태
    std::atomic<SessionState> state_{SessionState::CONNECTING};
    std::atomic<bool> isAuthenticated_{false};
    std::string userID_;
    
    // 위치
    std::atomic<int32_t> posX_{0};
    std::atomic<int32_t> posY_{0};
    
    // 버퍼
    RingBuffer recvBuffer_{8192};
    RingBuffer sendBuffer_{8192};
    
    // 송신 큐 (thread-safe)
    concurrency::concurrent_queue<PacketPtr> sendQueue_;
    std::atomic<bool> isSending_{false};
    
    // 작업 직렬화
    std::atomic<int32_t> isProcessing_{0};
    concurrency::concurrent_queue<Job> jobQueue_;
    
    // WSABUF
    WSABUF recvWsaBuf_{};
    WSABUF sendWsaBuf_{};
};

using SessionPtr = std::shared_ptr<Session>;
```

**Session.cpp**

```cpp
#include "Session.h"
#include "PacketDispatcher.h"
#include "SessionManager.h"
#include "Logger.h"
#include <MSWSock.h>

std::atomic<uint32_t> Session::s_nextSessionID{1};

constexpr size_t PACKET_HEADER_SIZE = 4;
constexpr size_t MAX_PACKET_SIZE = 8192;

Session::Session(SOCKET socket)
    : sessionID_(s_nextSessionID.fetch_add(1))
    , socket_(socket)
{
    LOG_INFO("세션 생성: SessionID={}", sessionID_);
}

Session::~Session()
{
    if (socket_ != INVALID_SOCKET)
    {
        closesocket(socket_);
        socket_ = INVALID_SOCKET;
    }
    
    LOG_INFO("세션 소멸: SessionID={}", sessionID_);
}

bool Session::RegisterRecv()
{
    if (!IsConnected())
        return false;
    
    // IOContext 생성 (shared_ptr 참조 포함)
    auto ioContext = new IOContext{};
    ioContext->session = shared_from_this();
    ioContext->ioType = IOType::RECV;
    
    // 수신 버퍼 준비
    recvWsaBuf_.buf = recvBuffer_.GetWriteBuffer();
    recvWsaBuf_.len = static_cast<ULONG>(recvBuffer_.GetFreeSize());
    
    DWORD flags = 0;
    DWORD bytes = 0;
    
    int result = WSARecv(socket_, &recvWsaBuf_, 1, &bytes, &flags,
                        &ioContext->overlapped, nullptr);
    
    if (result == SOCKET_ERROR)
    {
        int error = WSAGetLastError();
        if (error != WSA_IO_PENDING)
        {
            LOG_ERROR("WSARecv 실패: SessionID={}, Error={}", sessionID_, error);
            delete ioContext;
            Disconnect();
            return false;
        }
    }
    
    return true;
}

void Session::OnRecv(DWORD bytesTransferred)
{
    if (bytesTransferred == 0)
    {
        Disconnect();
        return;
    }
    
    // 작업을 큐에 추가하고 직렬 처리
    EnqueueJob([this, bytesTransferred]() {
        // 수신 버퍼 업데이트
        recvBuffer_.MoveWritePos(bytesTransferred);
        
        // 패킷 처리
        ProcessPackets();
        
        // 다음 수신 등록
        RegisterRecv();
    });
    
    TryProcessJobs();
}

void Session::ProcessPackets()
{
    // 완전한 패킷들 추출
    auto packets = ExtractPackets();
    
    // 각 패킷을 디스패처로 전달
    for (auto& packet : packets)
    {
        PacketDispatcher::Instance().Dispatch(shared_from_this(), packet);
    }
}

std::vector<PacketPtr> Session::ExtractPackets()
{
    std::vector<PacketPtr> packets;
    
    while (recvBuffer_.GetDataSize() >= PACKET_HEADER_SIZE)
    {
        // 패킷 크기 읽기 (Peek)
        uint16_t packetSize = 0;
        recvBuffer_.Peek(reinterpret_cast<char*>(&packetSize), sizeof(packetSize));
        
        // 패킷 크기 검증
        if (packetSize < PACKET_HEADER_SIZE || packetSize > MAX_PACKET_SIZE)
        {
            LOG_ERROR("잘못된 패킷 크기: Size={}, SessionID={}", 
                     packetSize, sessionID_);
            Disconnect();
            break;
        }
        
        // 완전한 패킷이 도착했는지 확인
        if (recvBuffer_.GetDataSize() < packetSize)
            break;
        
        // 완전한 패킷 추출
        auto packet = std::make_shared<Packet>(packetSize);
        size_t readSize = recvBuffer_.Read(packet->GetBuffer(), packetSize);
        
        if (readSize != packetSize)
        {
            LOG_ERROR("패킷 읽기 실패: Expected={}, Actual={}, SessionID={}",
                     packetSize, readSize, sessionID_);
            Disconnect();
            break;
        }
        
        packets.push_back(packet);
    }
    
    return packets;
}

void Session::Send(PacketPtr packet)
{
    // 송신 큐에 추가 (thread-safe)
    sendQueue_.push(packet);
    
    // 송신 시작 시도
    RegisterSend();
}

bool Session::RegisterSend()
{
    // 이미 송신 중이면 중복 등록 방지
    bool expected = false;
    if (!isSending_.compare_exchange_strong(expected, true))
        return false;
    
    // 송신 큐에서 패킷들 가져오기
    std::vector<PacketPtr> packets;
    PacketPtr packet;
    
    while (sendQueue_.try_pop(packet))
    {
        packets.push_back(packet);
        
        // 한 번에 최대 64개까지
        if (packets.size() >= 64)
            break;
    }
    
    if (packets.empty())
    {
        isSending_.store(false);
        return false;
    }
    
    // 송신 버퍼에 패킷들 복사
    sendBuffer_.Clear();
    for (auto& pkt : packets)
    {
        sendBuffer_.Write(pkt->GetBuffer(), pkt->GetSize());
    }
    
    // IOContext 생성
    auto ioContext = new IOContext{};
    ioContext->session = shared_from_this();
    ioContext->ioType = IOType::SEND;
    
    // WSABUF 설정
    sendWsaBuf_.buf = sendBuffer_.GetReadBuffer();
    sendWsaBuf_.len = static_cast<ULONG>(sendBuffer_.GetDataSize());
    
    DWORD bytes = 0;
    
    int result = WSASend(socket_, &sendWsaBuf_, 1, &bytes, 0,
                        &ioContext->overlapped, nullptr);
    
    if (result == SOCKET_ERROR)
    {
        int error = WSAGetLastError();
        if (error != WSA_IO_PENDING)
        {
            LOG_ERROR("WSASend 실패: SessionID={}, Error={}", sessionID_, error);
            delete ioContext;
            isSending_.store(false);
            Disconnect();
            return false;
        }
    }
    
    return true;
}

void Session::OnSend(DWORD bytesTransferred)
{
    // 송신 완료
    sendBuffer_.MoveReadPos(bytesTransferred);
    
    // 송신 플래그 해제
    isSending_.store(false);
    
    // 큐에 남은 패킷이 있으면 계속 송신
    if (!sendQueue_.empty())
    {
        RegisterSend();
    }
}

void Session::Disconnect()
{
    // 이미 연결 종료 중이면 무시
    if (!TryDisconnect())
        return;
    
    LOG_INFO("세션 연결 종료 시작: SessionID={}", sessionID_);
    
    // 소켓 종료
    if (socket_ != INVALID_SOCKET)
    {
        closesocket(socket_);
        socket_ = INVALID_SOCKET;
    }
    
    // 상태 변경
    state_.store(SessionState::DISCONNECTED, std::memory_order_release);
    
    // 정리 작업
    EnqueueJob([this]() {
        OnDisconnected();
    });
    
    TryProcessJobs();
}

void Session::OnDisconnected()
{
    LOG_INFO("세션 연결 종료 완료: SessionID={}, UserID={}", 
             sessionID_, userID_);
    
    // SessionManager에서 제거
    SessionManager::Instance().RemoveSession(sessionID_);
}

void Session::EnqueueJob(Job job)
{
    jobQueue_.push(std::move(job));
}

void Session::TryProcessJobs()
{
    // 처리 중이 아니면 처리 시작
    if (!TryBeginProcess())
        return;
    
    // 큐가 빌 때까지 처리
    while (true)
    {
        Job job;
        if (!jobQueue_.try_pop(job))
        {
            // 큐가 비었으면 종료
            EndProcess();
            
            // Double-Check: 끝내는 순간 추가된 작업 확인
            if (!jobQueue_.empty() && TryBeginProcess())
                continue;
            
            break;
        }
        
        try
        {
            job();  // 작업 실행
        }
        catch (const std::exception& e)
        {
            LOG_ERROR("작업 실행 예외: SessionID={}, Error={}", 
                     sessionID_, e.what());
        }
    }
}

bool Session::TryBeginProcess()
{
    int32_t expected = 0;
    return isProcessing_.compare_exchange_strong(
        expected, 1, std::memory_order_acq_rel
    );
}

void Session::EndProcess()
{
    isProcessing_.store(0, std::memory_order_release);
}
```

```cpp
void IOCPServer::WorkerThread()
{
    LOG_INFO("워커 스레드 시작: ThreadID={}", GetCurrentThreadId());
    
    while (running_)
    {
        DWORD bytesTransferred = 0;
        ULONG_PTR completionKey = 0;
        OVERLAPPED* overlapped = nullptr;
        
        BOOL result = GetQueuedCompletionStatus(
            iocpHandle_,
            &bytesTransferred,
            &completionKey,
            &overlapped,
            INFINITE
        );
        
        // 종료 신호
        if (completionKey == 0 && overlapped == nullptr)
        {
            break;
        }
        
        // IOContext 복원
        auto ioContext = reinterpret_cast<IOContext*>(overlapped);
        SessionPtr session = ioContext->session;
        
        // I/O 실패 처리
        if (!result || bytesTransferred == 0)
        {
            session->Disconnect();
            delete ioContext;
            continue;
        }
        
        // I/O 타입별 처리
        switch (ioContext->ioType)
        {
        case IOType::RECV:
            session->OnRecv(bytesTransferred);
            break;
            
        case IOType::SEND:
            session->OnSend(bytesTransferred);
            break;
        }
        
        // IOContext 정리 (shared_ptr 자동 해제)
        delete ioContext;
    }
    
    LOG_INFO("워커 스레드 종료: ThreadID={}", GetCurrentThreadId());
}
```

```cpp
        return instance;

    void BroadcastToAuthenticated(PacketPtr packet);
    
    // 모든 세션 순회
    template<typename Func>
    void ForEachSession(Func func);
    
private:
    SessionManager() = default;
    ~SessionManager() = default;
    
    // Concurrency Runtime의 thread-safe 컨테이너
    concurrency::concurrent_unordered_map<uint32_t, SessionPtr> sessions_;
};

template<typename Func>
void SessionManager::ForEachSession(Func func)
{
    for (auto& pair : sessions_)
    {
        if (pair.second)
        {
            func(pair.second);
        }
    }
}
```

```cpp
#include "SessionManager.h"
#include "Logger.h"

void SessionManager::AddSession(SessionPtr session)
{
    if (!session)
        return;
    
    uint32_t sessionID = session->GetID();
    sessions_[sessionID] = session;
    
    LOG_INFO("세션 추가: SessionID={}, TotalSessions={}",
             sessionID, sessions_.size());
}

void SessionManager::RemoveSession(uint32_t sessionID)
{
    auto it = sessions_.find(sessionID);
    if (it != sessions_.end())
    {
        sessions_.unsafe_erase(it);
        
        LOG_INFO("세션 제거: SessionID={}, TotalSessions={}",
                 sessionID, sessions_.size());
    }
}

SessionPtr SessionManager::GetSession(uint32_t sessionID)
{
    auto it = sessions_.find(sessionID);
    return (it != sessions_.end()) ? it->second : nullptr;
}

size_t SessionManager::GetSessionCount() const
{
    return sessions_.size();
}

void SessionManager::BroadcastPacket(PacketPtr packet)
{
    for (auto& pair : sessions_)
    {
        if (pair.second && pair.second->IsConnected())
        {
            pair.second->Send(packet);
        }
    }
}

void SessionManager::BroadcastPacket(PacketPtr packet, uint32_t excludeSessionID)
{
    for (auto& pair : sessions_)
    {
        if (pair.first != excludeSessionID && 
            pair.second && 
            pair.second->IsConnected())
        {
            pair.second->Send(packet);
        }
    }
}

void SessionManager::BroadcastToAuthenticated(PacketPtr packet)
{
    for (auto& pair : sessions_)
    {
        if (pair.second && 
            pair.second->IsConnected() && 
            pair.second->IsAuthenticated())
        {
            pair.second->Send(packet);
        }
    }
}
```

이제 우리는 멀티스레드 환경에서 안전하고 효율적으로 동작하는 IOCP 게임 서버를 구축했다. 주요 동기화 기법들:

1. **std::atomic**: 간단한 상태 변수와 카운터에 사용
2. **작업 직렬화**: 같은 세션의 작업은 순차적으로 처리
3. **shared_ptr**: 자동 수명 관리로 안전한 세션 참조
4. **concurrent 컨테이너**: 락 없는 thread-safe 컨테이너

다음 챕터에서는 Zero-Copy와 Scatter-Gather I/O를 통해 메모리 복사 오버헤드를 최소화하고 성능을 더욱 향상시키는 기법을 학습한다.
