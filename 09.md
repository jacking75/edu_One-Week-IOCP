# Chapter 9. 타이머와 하트비트 구현

게임 서버는 단순히 클라이언트의 요청에 응답하는 것만으로는 충분하지 않다. 주기적으로 몬스터를 생성하거나, 버프 효과를 만료시키거나, 연결이 끊긴 클라이언트를 감지하는 등 시간 기반의 작업들이 필수적이다. 이번 챕터에서는 IOCP 환경에서 효율적으로 동작하는 타이머 시스템을 구현하고, 클라이언트의 연결 상태를 확인하는 하트비트 메커니즘을 다룬다.

## 9.1 게임 서버에서의 타이머 필요성

**타이머가 필요한 상황들**

게임 서버에서 타이머는 다양한 목적으로 사용된다. 가장 기본적으로는 클라이언트의 연결 상태를 확인하는 하트비트 체크가 있다. 만약 클라이언트가 30초 동안 아무런 패킷도 보내지 않는다면, 네트워크 문제로 연결이 끊긴 것으로 간주하고 세션을 정리해야 한다. 또한 버프나 디버프 같은 시간 제한이 있는 효과들도 타이머로 관리된다. 10초 동안 공격력이 증가하는 버프를 받았다면, 정확히 10초 후에 효과를 제거해야 한다.

게임 로직 차원에서도 타이머는 중요하다. 몬스터를 5분마다 리스폰시키거나, 매일 자정에 일일 퀘스트를 초기화하거나, 1시간마다 게임 내 경제 시스템을 업데이트하는 등의 작업이 모두 타이머를 통해 이루어진다. 이러한 타이머들은 수십만 개가 동시에 동작할 수 있기 때문에, 효율적인 구현이 필수적이다.

**일반적인 타이머 구현의 문제점**

가장 단순한 타이머 구현은 별도의 스레드를 만들어서 Sleep()으로 대기한 후 작업을 수행하는 것이다. 하지만 이 방식은 타이머 하나당 스레드 하나가 필요하므로 확장성이 없다. 수천 개의 타이머를 위해 수천 개의 스레드를 만들 수는 없다.

또 다른 방법은 메인 루프에서 매 프레임마다 모든 타이머를 순회하며 시간이 지났는지 확인하는 것이다. 하지만 타이머가 많아지면 이 검사 자체가 큰 오버헤드가 된다. 10만 개의 타이머를 매번 체크하는 것은 비효율적이다.

```cpp
// 비효율적인 타이머 구현 예시
void CheckAllTimers() {
    auto now = std::chrono::steady_clock::now();
    
    // 모든 타이머를 매번 순회 - O(n) 복잡도
    for (auto& timer : allTimers) {
        if (now >= timer.expireTime) {
            timer.callback();
        }
    }
}
```

**IOCP 환경에서의 타이머 설계 방향**

IOCP 게임 서버에서 타이머는 두 가지 요구사항을 만족해야 한다. 첫째, 워커 스레드의 I/O 처리를 방해하지 않아야 한다. 타이머 체크 때문에 네트워크 패킷 처리가 지연되면 안 된다. 둘째, 대량의 타이머를 효율적으로 관리할 수 있어야 한다. 수만 개의 타이머가 동시에 동작해도 성능 저하가 없어야 한다.

이를 위해 우리는 타이머 전용 스레드를 하나 만들고, 만료된 타이머의 콜백을 IOCP 워커 스레드로 전달하는 방식을 사용한다. 타이머 자체는 별도 스레드에서 관리되지만, 실제 작업은 워커 스레드 풀에서 처리되므로 스레드 리소스를 절약할 수 있다.

## 9.2 Timer Queue와 Hierarchical Timing Wheel

**Timer Queue의 기본 개념**

타이머를 효율적으로 관리하기 위한 가장 기본적인 자료구조는 우선순위 큐(Priority Queue)다. 만료 시간이 가장 빠른 타이머를 힙의 루트에 두면, 다음에 처리할 타이머를 O(1)에 찾을 수 있다. 타이머 추가는 O(log n), 삭제도 O(log n)이므로 합리적인 성능을 보인다.

```
Timer Queue 구조 (Min Heap):

           [10:00:05]
          /          \
    [10:00:08]    [10:00:12]
    /        \
[10:00:15] [10:00:20]

→ 루트가 가장 빠르게 만료되는 타이머
```

C++ 표준 라이브러리의 `std::priority_queue`를 사용하면 간단히 구현할 수 있다. 하지만 게임 서버에서는 타이머를 중간에 취소하는 경우가 많은데, `priority_queue`는 중간 요소 삭제가 어렵다는 단점이 있다. 이를 해결하기 위해 타이머 ID를 부여하고, 취소된 타이머는 실행 시점에 유효성을 검사하는 방식을 사용한다.

**Hierarchical Timing Wheel 개념**

대량의 타이머를 더욱 효율적으로 처리하기 위한 고급 기법이 Timing Wheel이다. 시계처럼 여러 개의 슬롯을 두고, 각 슬롯에 해당 시간에 만료되는 타이머들을 저장하는 방식이다. 예를 들어 256개의 슬롯을 두고 각 슬롯이 1초를 담당한다면, 256초 이내의 모든 타이머를 O(1)에 추가하고 조회할 수 있다.

```
Timing Wheel 개념도:

        슬롯 0 (0초)  → [타이머A, 타이머B]
        슬롯 1 (1초)  → [타이머C]
        슬롯 2 (2초)  → []
        ...
        슬롯 255      → [타이머D]
        
→ 현재 시간의 슬롯만 검사하면 됨 (O(1))
```

Hierarchical Timing Wheel은 이를 확장하여 여러 단계의 휠을 사용한다. 초 단위 휠, 분 단위 휠, 시간 단위 휠을 계층적으로 구성하면 매우 긴 타이머도 효율적으로 관리할 수 있다. 하지만 구현 복잡도가 높아지므로, 이번 챕터에서는 우선순위 큐 기반의 Timer Queue를 구현하고, 최적화가 필요한 경우에만 Timing Wheel을 고려한다.

**Windows의 Timer Queue API**

Windows는 `CreateTimerQueue`, `CreateTimerQueueTimer` 같은 API를 제공한다. 이를 사용하면 커널 레벨에서 최적화된 타이머를 쉽게 사용할 수 있다. 하지만 게임 서버에서는 타이머와 게임 로직의 통합이 중요하므로, 직접 타이머 시스템을 구현하는 것이 더 나은 선택이다. 이를 통해 세션 객체와의 연동, 취소 로직, 에러 핸들링 등을 자유롭게 제어할 수 있다.

## 9.3 IOCP 기반 타이머 구현

**타이머 구조 설계**

먼저 타이머 하나를 표현하는 구조체를 정의한다. 각 타이머는 고유 ID, 만료 시간, 실행할 콜백, 그리고 반복 여부 정보를 가진다.

```cpp
struct TimerEvent {
    uint64_t timerId;           // 타이머 고유 ID
    std::chrono::steady_clock::time_point expireTime;  // 만료 시간
    std::function<void()> callback;  // 실행할 함수
    std::chrono::milliseconds interval;  // 반복 간격 (0이면 일회성)
    bool isCancelled;           // 취소 플래그
    
    // 우선순위 큐를 위한 비교 연산자
    bool operator>(const TimerEvent& other) const {
        return expireTime > other.expireTime;
    }
};
```

타이머 ID는 단순히 증가하는 카운터를 사용하되, 멀티스레드 환경이므로 `InterlockedIncrement64`로 원자적으로 증가시킨다. 취소 플래그는 타이머를 중간에 취소할 때 사용되며, 우선순위 큐에서 제거하지 않고 플래그만 설정하여 실행 시점에 필터링한다.

**TimerManager 클래스 구현**

TimerManager는 타이머 전용 스레드를 가지며, 우선순위 큐로 타이머를 관리한다. IOCP 워커 스레드와의 통합을 위해 CompletionPort 핸들을 받아둔다.

```cpp
class TimerManager {
private:
    std::priority_queue<TimerEvent, std::vector<TimerEvent>, 
                       std::greater<TimerEvent>> timerQueue_;
    std::unordered_map<uint64_t, bool> cancelledTimers_;
    
    std::mutex queueMutex_;
    std::condition_variable cv_;
    std::atomic<bool> isRunning_;
    std::thread timerThread_;
    
    HANDLE hCompletionPort_;  // IOCP 핸들
    std::atomic<uint64_t> nextTimerId_;

public:
    void Initialize(HANDLE hCompletionPort) {
        hCompletionPort_ = hCompletionPort;
        nextTimerId_ = 1;
        isRunning_ = true;
        timerThread_ = std::thread(&TimerManager::TimerThreadFunc, this);
    }
    
    uint64_t AddTimer(std::chrono::milliseconds delay, 
                     std::function<void()> callback,
                     std::chrono::milliseconds interval = std::chrono::milliseconds(0)) {
        uint64_t timerId = nextTimerId_.fetch_add(1);
        auto expireTime = std::chrono::steady_clock::now() + delay;
        
        TimerEvent event{timerId, expireTime, callback, interval, false};
        
        {
            std::lock_guard<std::mutex> lock(queueMutex_);
            timerQueue_.push(event);
        }
        cv_.notify_one();
        
        return timerId;
    }
};
```

타이머 추가는 간단하다. 만료 시간을 계산하여 우선순위 큐에 넣고, condition variable로 타이머 스레드를 깨운다. 만약 추가된 타이머가 기존의 가장 빠른 타이머보다 더 빨리 만료된다면, 타이머 스레드는 더 일찍 깨어나야 한다.

**타이머 스레드 구현**

타이머 스레드는 다음 타이머까지 대기했다가, 만료된 타이머의 콜백을 IOCP에 포스팅한다.

```cpp
void TimerThreadFunc() {
    while (isRunning_) {
        std::unique_lock<std::mutex> lock(queueMutex_);
        
        if (timerQueue_.empty()) {
            // 타이머가 없으면 대기
            cv_.wait(lock);
            continue;
        }
        
        auto now = std::chrono::steady_clock::now();
        const auto& nextTimer = timerQueue_.top();
        
        if (nextTimer.expireTime > now) {
            // 다음 타이머까지 대기
            cv_.wait_until(lock, nextTimer.expireTime);
            continue;
        }
        
        // 만료된 타이머 처리
        TimerEvent event = timerQueue_.top();
        timerQueue_.pop();
        lock.unlock();
        
        // 취소된 타이머는 무시
        if (event.isCancelled) continue;
        
        // IOCP에 타이머 작업 포스팅
        PostTimerCallback(event);
        
        // 반복 타이머면 다시 추가
        if (event.interval.count() > 0) {
            event.expireTime = std::chrono::steady_clock::now() + event.interval;
            lock.lock();
            timerQueue_.push(event);
        }
    }
}
```

이 구조의 핵심은 타이머 만료 처리를 IOCP 워커 스레드에 위임한다는 점이다. 타이머 스레드는 단순히 시간을 체크하고 작업을 포스팅하는 역할만 하므로, 콜백이 오래 걸려도 다른 타이머에 영향을 주지 않는다.

**IOCP 통합**

타이머 콜백을 IOCP 워커 스레드에서 실행하기 위해, 커스텀 Completion Key를 정의한다.

```cpp
enum class IOOperation : ULONG_PTR {
    Accept = 1,
    Receive = 2,
    Send = 3,
    Timer = 4  // 타이머 전용
};

struct TimerContext : public OVERLAPPED {
    IOOperation opType = IOOperation::Timer;
    std::function<void()>* pCallback;
};

void PostTimerCallback(const TimerEvent& event) {
    // 힙에 콜백 할당 (워커 스레드에서 해제)
    auto* pCallback = new std::function<void()>(event.callback);
    
    auto* timerCtx = new TimerContext();
    timerCtx->pCallback = pCallback;
    
    PostQueuedCompletionStatus(
        hCompletionPort_,
        0,  // bytes transferred
        static_cast<ULONG_PTR>(IOOperation::Timer),
        reinterpret_cast<LPOVERLAPPED>(timerCtx)
    );
}
```

워커 스레드에서는 `GetQueuedCompletionStatus`로 받은 Completion Key가 `IOOperation::Timer`일 때 타이머 콜백을 실행한다.

```cpp
void WorkerThread() {
    while (true) {
        DWORD bytesTransferred;
        ULONG_PTR completionKey;
        OVERLAPPED* pOverlapped;
        
        BOOL result = GetQueuedCompletionStatus(
            hCompletionPort_, &bytesTransferred, 
            &completionKey, &pOverlapped, INFINITE);
        
        auto opType = static_cast<IOOperation>(completionKey);
        
        if (opType == IOOperation::Timer) {
            auto* timerCtx = reinterpret_cast<TimerContext*>(pOverlapped);
            (*timerCtx->pCallback)();  // 콜백 실행
            delete timerCtx->pCallback;
            delete timerCtx;
        }
        // ... 다른 I/O 처리
    }
}
```

이렇게 하면 타이머 콜백이 워커 스레드 풀에서 병렬로 실행되면서도, 기존 I/O 처리 로직과 자연스럽게 통합된다.

## 9.4 하트비트(Ping/Pong) 시스템

**하트비트의 필요성**

네트워크 게임에서 가장 골치 아픈 문제 중 하나는 "유령 세션(Ghost Session)"이다. 클라이언트의 네트워크 케이블이 뽑히거나 강제 종료되면, 서버는 이를 즉시 알 수 없다. TCP의 Keep-Alive도 있지만 기본 설정으로는 2시간이나 걸린다. 이 시간 동안 서버는 끊어진 세션을 위해 메모리를 계속 점유하게 된다.

하트비트는 주기적으로 작은 패킷을 주고받아 연결 상태를 확인하는 메커니즘이다. 예를 들어 15초마다 서버가 Ping 패킷을 보내고, 클라이언트가 Pong 패킷으로 응답한다. 만약 30초 동안 응답이 없으면 연결이 끊긴 것으로 간주하고 세션을 정리한다.

```
정상적인 하트비트 흐름:

Server          Client
  |----Ping---->|  (0초)
  |<---Pong-----|
  |----Ping---->|  (15초)
  |<---Pong-----|
  |----Ping---->|  (30초)
  |<---Pong-----|

타임아웃 상황:

Server          Client
  |----Ping---->|  (0초)
  |<---Pong-----|
  |----Ping---->|  (15초)
  |   (응답 없음)
  |----Ping---->|  (30초)
  |   (응답 없음)
  [타임아웃 처리]
```

**하트비트 패킷 설계**

하트비트 패킷은 최대한 간단해야 한다. 불필요한 대역폭을 낭비하지 않으면서도 연결 상태를 확인할 수 있어야 한다.

```cpp
// 패킷 ID 추가
enum class PacketId : uint16_t {
    // 기존 패킷들...
    Ping = 100,
    Pong = 101,
};

struct PingPacket {
    PacketHeader header;
    uint64_t timestamp;  // 서버 시간 (RTT 측정용)
};

struct PongPacket {
    PacketHeader header;
    uint64_t timestamp;  // 받은 Ping의 timestamp 그대로 반환
};
```

Ping 패킷에 타임스탬프를 넣으면 RTT(Round Trip Time)도 측정할 수 있다. 이는 네트워크 레이턴시 모니터링이나 디버깅에 유용하다.

**세션에 하트비트 정보 추가**

각 세션은 마지막으로 데이터를 받은 시간을 기록해야 한다. 타이머 매니저와 연동하여 주기적으로 타임아웃을 체크한다.

```cpp
class Session {
private:
    std::atomic<std::chrono::steady_clock::time_point> lastRecvTime_;
    std::atomic<std::chrono::steady_clock::time_point> lastPingTime_;
    std::chrono::seconds heartbeatTimeout_;  // 예: 30초
    uint64_t heartbeatTimerId_;
    
public:
    void OnReceive(const char* data, int len) {
        // 어떤 패킷이든 받으면 마지막 수신 시간 업데이트
        lastRecvTime_.store(std::chrono::steady_clock::now());
        
        // 패킷 처리...
    }
    
    void StartHeartbeat(TimerManager* pTimerMgr) {
        // 15초마다 Ping 전송
        heartbeatTimerId_ = pTimerMgr->AddTimer(
            std::chrono::seconds(15),
            [this]() { this->SendPing(); },
            std::chrono::seconds(15)  // 반복
        );
        
        // 타임아웃 체크 타이머 (5초마다 체크)
        pTimerMgr->AddTimer(
            std::chrono::seconds(5),
            [this]() { this->CheckHeartbeatTimeout(); },
            std::chrono::seconds(5)
        );
    }
};
```

이렇게 하면 클라이언트가 Pong 패킷뿐 아니라 다른 패킷을 보내도 "살아있음"으로 간주된다. 실제 게임 중에는 이동이나 공격 패킷이 계속 오므로, Ping/Pong이 거의 발생하지 않을 수 있다.

**Ping 전송과 Pong 처리**

서버는 주기적으로 Ping을 보내고, 클라이언트는 즉시 Pong으로 응답한다.

## 9.5 타임아웃 검출과 자동 연결 해제

**타임아웃 검출 로직**

타임아웃 검출은 주기적으로 마지막 수신 시간을 체크하여 임계값을 넘었는지 확인한다.

```cpp
void Session::CheckHeartbeatTimeout() {
    auto now = std::chrono::steady_clock::now();
    auto lastRecv = lastRecvTime_.load();
    
    auto elapsed = std::chrono::duration_cast<std::chrono::seconds>(
        now - lastRecv);
    
    if (elapsed > heartbeatTimeout_) {
        std::cout << "Session " << sessionId_ 
                  << " heartbeat timeout (" << elapsed.count() 
                  << "s). Disconnecting...\n";
        
        // 강제 종료
        ForceDisconnect(DisconnectReason::Timeout);
    }
}
```

이 함수는 타이머 매니저에 의해 주기적으로 호출된다. 타임아웃이 감지되면 즉시 세션을 종료하여 리소스를 회수한다.

**안전한 연결 해제 처리**

타임아웃으로 인한 강제 종료는 일반적인 종료보다 더 조심스럽게 처리해야 한다. 세션이 이미 종료 중이거나, 동시에 여러 타이머에서 종료를 시도할 수 있기 때문이다.

```cpp
enum class SessionState {
    Connected,
    Disconnecting,
    Disconnected
};

void Session::ForceDisconnect(DisconnectReason reason) {
    // 상태를 원자적으로 변경
    SessionState expected = SessionState::Connected;
    if (!state_.compare_exchange_strong(expected, SessionState::Disconnecting)) {
        // 이미 종료 중이거나 종료됨
        return;
    }
    
    // 모든 타이머 취소
    if (heartbeatTimerId_ != 0) {
        pTimerManager_->CancelTimer(heartbeatTimerId_);
    }
    
    // 소켓 종료
    if (socket_ != INVALID_SOCKET) {
        closesocket(socket_);
        socket_ = INVALID_SOCKET;
    }
    
    // 세션 매니저에 정리 요청
    pSessionManager_->RemoveSession(sessionId_);
    
    state_.store(SessionState::Disconnected);
}
```

`compare_exchange_strong`을 사용하여 상태 변경을 원자적으로 수행한다. 이미 다른 스레드가 종료 처리를 시작했다면, 중복 처리를 방지하고 즉시 리턴한다.

**타이머 취소 구현**

타이머를 중간에 취소하는 기능도 필요하다. 세션이 정상 종료될 때 하트비트 타이머를 중지해야 한다.

```cpp
class TimerManager {
public:
    void CancelTimer(uint64_t timerId) {
        std::lock_guard<std::mutex> lock(queueMutex_);
        cancelledTimers_[timerId] = true;
        // 우선순위 큐에서 직접 제거하지 않고, 
        // 실행 시점에 플래그로 필터링
    }
};
```

우선순위 큐에서 중간 요소를 제거하는 것은 비효율적이므로, 취소 플래그를 설정하고 실행 시점에 걸러내는 lazy deletion 방식을 사용한다. 만약 취소된 타이머가 너무 많이 쌓이면 주기적으로 큐를 재구성할 수도 있다.

**그레이스풀 셧다운(Graceful Shutdown)**

서버가 종료될 때는 모든 타이머를 정리하고 스레드를 안전하게 종료해야 한다.

```cpp
class TimerManager {
public:
    void Shutdown() {
        isRunning_ = false;
        cv_.notify_all();
        
        if (timerThread_.joinable()) {
            timerThread_.join();
        }
        
        // 남은 타이머 정리
        std::lock_guard<std::mutex> lock(queueMutex_);
        while (!timerQueue_.empty()) {
            timerQueue_.pop();
        }
        cancelledTimers_.clear();
    }
};
```

서버의 메인 스레드는 종료 신호를 받으면 각 매니저의 `Shutdown()`을 순서대로 호출하여 깨끗하게 종료한다.

**TimerManager.h**

```cpp
#pragma once
#include <windows.h>
#include <functional>
#include <queue>
#include <vector>
#include <unordered_map>
#include <thread>
#include <mutex>
#include <condition_variable>

struct TimerEvent {
    uint64_t timerId;
    std::chrono::steady_clock::time_point expireTime;
    std::function<void()> callback;
    std::chrono::milliseconds interval;
    bool isCancelled;
    
    bool operator>(const TimerEvent& other) const {
        return expireTime > other.expireTime;
    }
};

enum class IOOperation : ULONG_PTR {
    Accept = 1,
    Receive = 2,
    Send = 3,
    Timer = 4
};

struct TimerContext : public OVERLAPPED {
    IOOperation opType = IOOperation::Timer;
    std::function<void()>* pCallback;
};

class TimerManager {
private:
    std::priority_queue<TimerEvent, std::vector<TimerEvent>, 
                       std::greater<TimerEvent>> timerQueue_;
    std::unordered_map<uint64_t, bool> cancelledTimers_;
    
    std::mutex queueMutex_;
    std::condition_variable cv_;
    std::atomic<bool> isRunning_;
    std::thread timerThread_;
    
    HANDLE hCompletionPort_;
    std::atomic<uint64_t> nextTimerId_;

public:
    TimerManager() : hCompletionPort_(NULL), nextTimerId_(1), isRunning_(false) {}
    ~TimerManager() { Shutdown(); }
    
    void Initialize(HANDLE hCompletionPort);
    void Shutdown();
    
    uint64_t AddTimer(std::chrono::milliseconds delay, 
                     std::function<void()> callback,
                     std::chrono::milliseconds interval = std::chrono::milliseconds(0));
    
    void CancelTimer(uint64_t timerId);

private:
    void TimerThreadFunc();
    void PostTimerCallback(const TimerEvent& event);
};
```

**TimerManager.cpp**

```cpp
#include "TimerManager.h"
#include <iostream>

void TimerManager::Initialize(HANDLE hCompletionPort) {
    hCompletionPort_ = hCompletionPort;
    nextTimerId_ = 1;
    isRunning_ = true;
    timerThread_ = std::thread(&TimerManager::TimerThreadFunc, this);
}

void TimerManager::Shutdown() {
    if (!isRunning_) return;
    
    isRunning_ = false;
    cv_.notify_all();
    
    if (timerThread_.joinable()) {
        timerThread_.join();
    }
    
    std::lock_guard<std::mutex> lock(queueMutex_);
    while (!timerQueue_.empty()) {
        timerQueue_.pop();
    }
    cancelledTimers_.clear();
    
    std::cout << "TimerManager shut down.\n";
}

uint64_t TimerManager::AddTimer(std::chrono::milliseconds delay, 
                                std::function<void()> callback,
                                std::chrono::milliseconds interval) {
    if (!isRunning_) return 0;
    
    uint64_t timerId = nextTimerId_.fetch_add(1, std::memory_order_relaxed);
    auto expireTime = std::chrono::steady_clock::now() + delay;
    
    TimerEvent event{timerId, expireTime, callback, interval, false};
    
    {
        std::lock_guard<std::mutex> lock(queueMutex_);
        timerQueue_.push(event);
    }
    cv_.notify_one();
    
    return timerId;
}

void TimerManager::CancelTimer(uint64_t timerId) {
    std::lock_guard<std::mutex> lock(queueMutex_);
    cancelledTimers_[timerId] = true;
}

void TimerManager::TimerThreadFunc() {
    while (isRunning_) {
        std::unique_lock<std::mutex> lock(queueMutex_);
        
        // 큐가 비어있으면 대기
        if (timerQueue_.empty()) {
            cv_.wait(lock, [this] { 
                return !isRunning_ || !timerQueue_.empty(); 
            });
            
            if (!isRunning_) break;
            continue;
        }
        
        auto now = std::chrono::steady_clock::now();
        const auto& nextTimer = timerQueue_.top();
        
        // 다음 타이머가 아직 만료되지 않았으면 대기
        if (nextTimer.expireTime > now) {
            cv_.wait_until(lock, nextTimer.expireTime, [this] { 
                return !isRunning_; 
            });
            
            if (!isRunning_) break;
            continue;
        }
        
        // 만료된 타이머 처리
        TimerEvent event = timerQueue_.top();
        timerQueue_.pop();
        lock.unlock();
        
        // 취소된 타이머는 무시
        if (cancelledTimers_.count(event.timerId) > 0) {
            continue;
        }
        
        // IOCP에 타이머 작업 포스팅
        PostTimerCallback(event);
        
        // 반복 타이머면 다시 추가
        if (event.interval.count() > 0) {
            event.expireTime = std::chrono::steady_clock::now() + event.interval;
            lock.lock();
            timerQueue_.push(event);
        }
    }
    
    std::cout << "Timer thread exiting.\n";
}

void TimerManager::PostTimerCallback(const TimerEvent& event) {
    // 힙에 콜백 할당 (워커 스레드에서 해제)
    auto* pCallback = new std::function<void()>(event.callback);
    
    auto* timerCtx = new TimerContext();
    ZeroMemory(timerCtx, sizeof(OVERLAPPED));
    timerCtx->opType = IOOperation::Timer;
    timerCtx->pCallback = pCallback;
    
    BOOL result = PostQueuedCompletionStatus(
        hCompletionPort_,
        0,
        static_cast<ULONG_PTR>(IOOperation::Timer),
        reinterpret_cast<LPOVERLAPPED>(timerCtx)
    );
    
    if (!result) {
        std::cerr << "PostQueuedCompletionStatus failed: " 
                  << GetLastError() << "\n";
        delete pCallback;
        delete timerCtx;
    }
}

void NetworkThread() {

        char recvBuffer[4096];
        int recvLen = recv(socket_, recvBuffer, sizeof(recvBuffer), 0);
        
        if (recvLen <= 0) {
            // 연결 종료 또는 에러

        }

        int offset = 0;
        while (offset < recvLen) {
            PacketHeader* header = reinterpret_cast<PacketHeader*>(recvBuffer + offset);
            
            if (header->packetId == PacketId::Ping) {
                // 즉시 Pong 응답
                PingPacket* pingPacket = reinterpret_cast<PingPacket*>(recvBuffer + offset);
                
                PongPacket pong;
                pong.header.packetId = PacketId::Pong;
                pong.header.size = sizeof(PongPacket);
                pong.timestamp = pingPacket->timestamp;
                
                send(socket_, reinterpret_cast<char*>(&pong), sizeof(pong), 0);
            }
            else {
                // 다른 패킷 처리...
            }
            
            offset += header->size;
        }
    }
}
```

이 코드는 Chapter 9의 핵심 개념을 모두 포함하고 있다. TimerManager는 우선순위 큐로 타이머를 관리하고, 만료된 타이머를 IOCP를 통해 워커 스레드에 전달한다. Session은 하트비트 타이머를 활용하여 주기적으로 Ping을 보내고, 타임아웃을 감지하여 자동으로 연결을 해제한다. 클라이언트는 Ping을 받으면 즉시 Pong으로 응답하여 연결이 살아있음을 알린다.

다음 챕터에서는 IOCP 워커 스레드와 게임 로직 스레드를 분리하여, I/O 처리와 비즈니스 로직이 서로 영향을 주지 않도록 아키텍처를 개선한다.
