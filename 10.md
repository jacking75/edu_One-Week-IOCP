# Chapter 10. 게임 로직 스레드 분리

IOCP 워커 스레드는 네트워크 I/O를 최대한 빠르게 처리하는 것이 목적이다. 하지만 게임 로직이 복잡해지면 워커 스레드에서 직접 처리하기 어려워진다. 몬스터 AI 계산, 경로 탐색, 물리 시뮬레이션 같은 무거운 연산이 워커 스레드를 점유하면 다른 세션의 패킷 처리가 지연된다. 이번 챕터에서는 I/O 처리와 게임 로직을 명확히 분리하고, 작업 큐를 통해 안전하게 통신하는 아키�ecture를 구현한다.

## 10.1 I/O 스레드와 로직 스레드 분리의 필요성

**IOCP 워커 스레드의 역할과 한계**

IOCP 워커 스레드는 네트워크 이벤트를 처리하는 전문가다. `GetQueuedCompletionStatus`로 완료 통지를 받아 빠르게 처리하고 다음 I/O 작업을 준비한다. 이상적으로는 워커 스레드가 CPU를 오래 점유하지 않아야 한다. 마이크로초 단위로 작업을 끝내고 다음 완료 통지를 기다리는 것이 바람직하다.

하지만 게임 로직은 예측 불가능하다. 플레이어가 아이템을 사용하면 인벤토리 검증, 스킬 쿨다운 체크, 효과 적용, 주변 플레이어에게 브로드캐스트 등 여러 단계를 거친다. 이 과정이 수 밀리초에서 수십 밀리초까지 걸릴 수 있다. 워커 스레드 하나가 이런 작업에 묶여 있으면, 그 시간 동안 다른 세션의 패킷은 처리되지 못한다.

```
워커 스레드가 게임 로직을 직접 처리하는 경우:

Thread 1: [Recv A] → [Game Logic A (20ms)] → [Recv B] → ...
Thread 2: [Recv C] → [Recv D] → [Recv E] → ...

→ Thread 1이 로직 처리 중일 때 A의 다른 패킷은 대기
→ I/O 처리량 저하
```

**분리 아키텍처의 장점**

I/O 스레드와 로직 스레드를 분리하면 각자의 역할에 집중할 수 있다. I/O 스레드는 패킷을 받아서 파싱하고 작업 큐에 넣기만 한다. 로직 스레드는 큐에서 작업을 꺼내 게임 로직을 수행하고, 응답이 필요하면 다시 I/O 스레드에 전달한다.

```
분리된 아키텍처:

I/O Thread 1: [Recv A] → [Enqueue] → [Recv B] → [Enqueue] → ...
I/O Thread 2: [Recv C] → [Enqueue] → [Recv D] → [Enqueue] → ...

Logic Thread 1: [Dequeue] → [Process Game Logic] → [Enqueue Response] → ...
Logic Thread 2: [Dequeue] → [Process Game Logic] → [Enqueue Response] → ...

→ I/O 처리와 로직 처리가 독립적으로 동작
→ 각 스레드가 최적화된 작업에만 집중
```

이런 구조는 확장성도 좋다. 게임이 CPU 집약적이면 로직 스레드를 늘리고, 네트워크가 병목이면 I/O 스레드를 늘린다. 각 부분을 독립적으로 튜닝할 수 있다.

**Actor 모델과의 유사성**

이 아키텍처는 Actor 모델과 유사하다. 각 게임 객체(플레이어, 몬스터, 룸)를 독립적인 Actor로 보고, 메시지 큐를 통해서만 통신한다. Actor는 한 번에 하나의 메시지만 처리하므로 동기화 문제가 없다. 이는 게임 서버에서 매우 효과적인 패턴이다.

```
Actor 모델 개념:

┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Player A   │      │  Player B   │      │   Monster   │
│   Actor     │◄────►│   Actor     │◄────►│    Actor    │
│  [Queue]    │      │  [Queue]    │      │  [Queue]    │
└─────────────┘      └─────────────┘      └─────────────┘
       ▲                    ▲                    ▲
       │                    │                    │
       └────────────────────┴────────────────────┘
                    Logic Thread Pool
```

## 10.2 작업 큐(Job Queue) 설계

**Job 구조 정의**

작업 큐에 들어갈 Job은 실행할 함수와 필요한 컨텍스트를 담고 있어야 한다. 가장 간단한 방법은 `std::function`을 사용하는 것이다.

```cpp
struct Job {
    std::function<void()> task;
    uint64_t sessionId;  // 세션별 순서 보장을 위한 ID
    uint64_t sequenceNumber;  // 작업 순서
    
    Job(std::function<void()> t, uint64_t sid, uint64_t seq)
        : task(std::move(t)), sessionId(sid), sequenceNumber(seq) {}
};
```

sessionId를 저장하는 이유는 같은 세션의 작업들은 순서가 보장되어야 하기 때문이다. 플레이어가 "아이템 구매" 후 "아이템 사용" 패킷을 보냈다면, 반드시 그 순서대로 처리되어야 한다. 여러 로직 스레드가 있어도 같은 세션의 작업은 순차적으로 실행되어야 한다.

**Thread-Safe Queue 구현**

멀티 프로듀서(I/O 스레드들), 멀티 컨슈머(로직 스레드들) 환경이므로 스레드 안전한 큐가 필요하다. Concurrency Runtime의 `concurrent_queue`를 사용한다.

```cpp
#include <concurrent_queue.h>

class JobQueue {
private:
    concurrency::concurrent_queue<Job> queue_;
    std::atomic<uint64_t> totalEnqueued_;
    std::atomic<uint64_t> totalDequeued_;
    
public:
    JobQueue() : totalEnqueued_(0), totalDequeued_(0) {}
    
    void Enqueue(Job job) {
        queue_.push(std::move(job));
        totalEnqueued_.fetch_add(1, std::memory_order_relaxed);
    }
    
    bool TryDequeue(Job& outJob) {
        bool success = queue_.try_pop(outJob);
        if (success) {
            totalDequeued_.fetch_add(1, std::memory_order_relaxed);
        }
        return success;
    }
    
    size_t GetPendingCount() const {
        return totalEnqueued_.load(std::memory_order_relaxed) - 
               totalDequeued_.load(std::memory_order_relaxed);
    }
};
```

`concurrent_queue`는 lock-free로 구현되어 있어 성능이 우수하다. 하지만 완전한 대기 기능은 없으므로, 로직 스레드가 큐가 비었을 때 대기하려면 추가 동기화가 필요하다.

**세마포어를 이용한 대기 메커니즘**

C++20의 `std::counting_semaphore`를 사용하면 로직 스레드가 효율적으로 대기할 수 있다.

```cpp
#include <semaphore>

class JobQueue {
private:
    concurrency::concurrent_queue<Job> queue_;
    std::counting_semaphore<> semaphore_{0};  // C++20
    std::atomic<bool> isShutdown_;
    
public:
    void Enqueue(Job job) {
        queue_.push(std::move(job));
        semaphore_.release();  // 대기 중인 스레드 하나를 깨움
    }
    
    bool Dequeue(Job& outJob) {
        semaphore_.acquire();  // Job이 들어올 때까지 대기
        
        if (isShutdown_.load()) {
            return false;  // 종료 신호
        }
        
        return queue_.try_pop(outJob);
    }
    
    void Shutdown() {
        isShutdown_.store(true);
        // 모든 대기 중인 스레드를 깨움
        for (int i = 0; i < 100; ++i) {  // 충분한 수
            semaphore_.release();
        }
    }
};
```

이제 로직 스레드는 큐가 비어있으면 CPU를 소비하지 않고 대기하다가, Job이 들어오면 즉시 깨어나서 처리한다.

**우선순위 큐 고려사항**

게임 서버에서는 어떤 작업이 다른 작업보다 중요할 수 있다. 예를 들어 전투 관련 패킷은 채팅보다 우선 처리되어야 한다. 이런 경우 우선순위 큐를 고려할 수 있다.

```cpp
struct PriorityJob {
    Job job;
    int priority;  // 높을수록 우선
    
    bool operator<(const PriorityJob& other) const {
        return priority < other.priority;  // 우선순위 큐는 역순
    }
};
```

하지만 우선순위 큐는 lock-free 구현이 어렵고, 우선순위가 낮은 작업이 starvation에 빠질 수 있다. 일반적인 게임 서버에서는 FIFO 큐로도 충분하며, 정말 긴급한 작업은 별도의 고속 큐를 두는 것이 낫다.

## 10.3 게임 로직 스레드 구현

**LogicWorker 클래스 설계**

로직 워커는 Job을 큐에서 꺼내 실행하는 단순한 스레드다. 하지만 세션별 순서 보장, 에러 핸들링, 성능 모니터링 같은 추가 기능이 필요하다.

```cpp
class LogicWorker {
private:
    std::thread thread_;
    JobQueue* pJobQueue_;
    std::atomic<bool> isRunning_;
    
    // 세션별 마지막 처리 시퀀스 (순서 보장용)
    std::unordered_map<uint64_t, uint64_t> sessionLastSequence_;
    std::mutex sessionSeqMutex_;
    
    // 통계
    std::atomic<uint64_t> processedJobCount_;
    std::atomic<uint64_t> totalProcessingTimeMs_;

public:
    LogicWorker(JobQueue* pQueue) 
        : pJobQueue_(pQueue)
        , isRunning_(false)
        , processedJobCount_(0)
        , totalProcessingTimeMs_(0) {}
    
    void Start() {
        isRunning_ = true;
        thread_ = std::thread(&LogicWorker::WorkerThreadFunc, this);
    }
    
    void Stop() {
        isRunning_ = false;
        if (thread_.joinable()) {
            thread_.join();
        }
    }
    
    uint64_t GetProcessedJobCount() const {
        return processedJobCount_.load(std::memory_order_relaxed);
    }
    
    double GetAverageProcessingTimeMs() const {
        uint64_t count = processedJobCount_.load(std::memory_order_relaxed);
        if (count == 0) return 0.0;
        return static_cast<double>(totalProcessingTimeMs_.load(std::memory_order_relaxed)) / count;
    }

private:
    void WorkerThreadFunc();
    bool ShouldProcessJob(const Job& job);
};
```

통계 정보를 수집하면 어떤 작업이 얼마나 시간을 소비하는지 분석할 수 있다. 이는 성능 튜닝에 매우 유용하다.

**워커 스레드 메인 루프**

워커 스레드는 무한 루프로 Job을 처리한다. 예외 처리와 순서 보장 로직이 포함된다.

```cpp
void LogicWorker::WorkerThreadFunc() {
    while (isRunning_) {
        Job job;
        
        // Job 대기 및 가져오기
        if (!pJobQueue_->Dequeue(job)) {
            // 종료 신호
            break;
        }
        
        // 세션별 순서 보장 체크
        if (!ShouldProcessJob(job)) {
            // 순서가 맞지 않으면 다시 큐에 넣기
            pJobQueue_->Enqueue(std::move(job));
            std::this_thread::sleep_for(std::chrono::milliseconds(1));
            continue;
        }
        
        // Job 실행
        auto startTime = std::chrono::steady_clock::now();
        
        try {
            job.task();
        }
        catch (const std::exception& e) {
            std::cerr << "[LogicWorker] Exception in job processing: " 
                      << e.what() << "\n";
        }
        catch (...) {
            std::cerr << "[LogicWorker] Unknown exception in job processing\n";
        }
        
        auto endTime = std::chrono::steady_clock::now();
        auto elapsedMs = std::chrono::duration_cast<std::chrono::milliseconds>(
            endTime - startTime).count();
        
        // 통계 업데이트
        processedJobCount_.fetch_add(1, std::memory_order_relaxed);
        totalProcessingTimeMs_.fetch_add(elapsedMs, std::memory_order_relaxed);
        
        // 너무 오래 걸린 작업 경고
        if (elapsedMs > 100) {
            std::cout << "[LogicWorker] Long job detected: " 
                      << elapsedMs << "ms\n";
        }
    }
    
    std::cout << "[LogicWorker] Thread exiting.\n";
}
```

예외가 발생해도 워커 스레드가 종료되지 않도록 try-catch로 감싼다. 게임 로직 버그로 서버 전체가 다운되는 것을 방지한다.

**세션별 순서 보장 로직**

같은 세션의 Job들은 순서대로 처리되어야 한다. 시퀀스 번호를 사용하여 이를 보장한다.

```cpp
bool LogicWorker::ShouldProcessJob(const Job& job) {
    std::lock_guard<std::mutex> lock(sessionSeqMutex_);
    
    auto it = sessionLastSequence_.find(job.sessionId);
    
    if (it == sessionLastSequence_.end()) {
        // 이 세션의 첫 Job
        sessionLastSequence_[job.sessionId] = job.sequenceNumber;
        return true;
    }
    
    // 다음 시퀀스가 맞는지 확인
    if (job.sequenceNumber == it->second + 1) {
        it->second = job.sequenceNumber;
        return true;
    }
    
    // 순서가 맞지 않음
    return false;
}
```

이 방식의 단점은 한 세션의 작업이 지연되면 해당 세션의 모든 후속 작업도 지연된다는 것이다. 더 정교한 구현은 세션별로 별도의 큐를 두고, 로직 스레드가 여러 세션의 큐를 순회하며 처리하는 방식이다. 하지만 구현 복잡도가 높아지므로, 이 책에서는 단순한 방식을 사용한다.

## 10.4 스레드 간 메시지 전달

**I/O 스레드에서 로직 스레드로**

I/O 스레드(IOCP 워커)는 패킷을 받으면 Job으로 변환하여 큐에 넣는다.

```cpp
void Session::OnPacketReceived(PacketHeader* pHeader, char* pData) {
    // 패킷 파싱
    auto packetId = pHeader->packetId;
    
    // Job 생성 및 큐에 추가
    uint64_t seq = GetNextSequenceNumber();  // 세션별 증가 카운터
    
    Job job([this, packetId, pData, size = pHeader->size]() {
        // 로직 스레드에서 실행될 코드
        this->ProcessGameLogic(packetId, pData, size);
    }, this->sessionId_, seq);
    
    pJobQueue_->Enqueue(std::move(job));
}
```

람다 캡처를 사용하여 필요한 데이터를 Job에 담는다. `this` 포인터를 캡처하므로 세션의 생명 주기 관리에 주의해야 한다. 7장에서 다룬 참조 카운팅을 활용한다.

```cpp
void Session::OnPacketReceived(PacketHeader* pHeader, char* pData) {
    // 참조 카운트 증가 (Job이 실행될 때까지 세션 유지)
    AddRef();
    
    auto packetId = pHeader->packetId;
    uint64_t seq = GetNextSequenceNumber();
    
    // 패킷 데이터 복사 (I/O 버퍼가 재사용될 수 있으므로)
    std::vector<char> packetCopy(pData, pData + pHeader->size);
    
    Job job([this, packetId, data = std::move(packetCopy)]() {
        this->ProcessGameLogic(packetId, data.data(), data.size());
        this->Release();  // Job 완료 후 참조 카운트 감소
    }, sessionId_, seq);
    
    pJobQueue_->Enqueue(std::move(job));
}
```

**로직 스레드에서 I/O 스레드로**

로직 처리 후 응답 패킷을 보내야 할 때는 다시 I/O 스레드로 전달한다. 직접 `Send()`를 호출해도 되지만, 명확한 분리를 위해 별도의 응답 큐를 사용할 수도 있다.

```cpp
void Session::ProcessGameLogic(PacketId packetId, const char* pData, int size) {
    // 게임 로직 처리
    if (packetId == PacketId::MoveRequest) {
        // 이동 처리...
        
        // 응답 패킷 생성
        MoveResponsePacket response;
        response.header.packetId = PacketId::MoveResponse;
        response.header.size = sizeof(MoveResponsePacket);
        // ... 응답 데이터 채우기
        
        // 전송 (Send는 내부적으로 WSASend를 비동기로 호출)
        Send(reinterpret_cast<char*>(&response), sizeof(response));
    }
}
```

`Send()` 함수는 송신 버퍼에 데이터를 넣고 `WSASend`를 호출한다. 이는 비동기 작업이므로 로직 스레드를 블록하지 않는다. 실제 전송 완료는 IOCP 워커 스레드에서 처리된다.

**브로드캐스트 최적화**

한 플레이어의 행동을 주변 플레이어들에게 알려야 할 때, 수백 명에게 개별적으로 패킷을 보내면 비효율적이다. 패킷을 한 번만 직렬화하고 여러 세션에 전송하는 최적화가 필요하다.

```cpp
class Room {
public:
    void BroadcastPacket(const char* pData, int size, uint64_t exceptSessionId = 0) {
        // 패킷을 한 번만 준비
        std::vector<char> packetData(pData, pData + size);
        
        // 모든 세션에 전송
        for (auto& session : sessions_) {
            if (session->GetSessionId() == exceptSessionId) continue;
            
            // 각 세션의 송신 큐에 추가 (복사 최소화)
            session->SendShared(packetData);
        }
    }
};
```

`SendShared`는 `shared_ptr`나 참조 카운팅을 사용하여 같은 데이터를 여러 세션이 공유하도록 한다. Zero-Copy 기법과 결합하면 더욱 효율적이다.

## 10.5 로직 처리 순서 보장

**같은 세션 내 순서 보장**

앞서 시퀀스 번호로 순서를 보장하는 방법을 봤다. 하지만 이 방식은 순서가 어긋난 Job을 다시 큐에 넣으므로 비효율적이다. 더 나은 방법은 세션별로 pending queue를 두는 것이다.

```cpp
class Session {
private:
    std::queue<Job> pendingJobs_;
    std::mutex pendingMutex_;
    uint64_t expectedSequence_;
    std::atomic<bool> isProcessing_;

public:
    void EnqueueJob(Job job) {
        std::lock_guard<std::mutex> lock(pendingMutex_);
        pendingJobs_.push(std::move(job));
        
        // 처리 중이 아니면 로직 큐에 첫 Job을 넣음
        bool expected = false;
        if (isProcessing_.compare_exchange_strong(expected, true)) {
            FlushPendingJobs();
        }
    }
    
private:
    void FlushPendingJobs() {
        std::lock_guard<std::mutex> lock(pendingMutex_);
        
        while (!pendingJobs_.empty()) {
            Job& job = pendingJobs_.front();
            
            if (job.sequenceNumber != expectedSequence_) {
                // 순서가 맞지 않으면 중단
                break;
            }
            
            // 로직 큐에 추가
            pJobQueue_->Enqueue(std::move(job));
            pendingJobs_.pop();
            expectedSequence_++;
        }
        
        if (pendingJobs_.empty()) {
            isProcessing_.store(false);
        }
    }
};
```

이 방식은 Job이 순서대로 들어오면 즉시 처리되고, 순서가 어긋나면 pending queue에서 대기한다. 하지만 구현이 복잡하고 세션마다 별도의 큐를 관리해야 하는 단점이 있다.

**실용적인 접근: 단일 스레드 Room**

실제 게임 서버에서는 Room 단위로 단일 스레드를 할당하는 경우가 많다. Room 안의 모든 작업은 하나의 스레드에서 순차적으로 처리되므로 동기화 문제가 없다.

```
Room-based Threading:

┌──────────────────────────────────────────┐
│              I/O Thread Pool              │
│  [Recv] → [Parse] → [Enqueue to Room]    │
└──────────────────┬───────────────────────┘
                   │
         ┌─────────┴─────────┬──────────────┐
         ▼                   ▼              ▼
    ┌─────────┐         ┌─────────┐    ┌─────────┐
    │ Room 1  │         │ Room 2  │    │ Room 3  │
    │ Thread  │         │ Thread  │    │ Thread  │
    │ [Queue] │         │ [Queue] │    │ [Queue] │
    └─────────┘         └─────────┘    └─────────┘
```

각 Room은 자신만의 스레드와 Job Queue를 가진다. Room 내 플레이어들의 패킷은 모두 그 Room의 큐로 전달되므로, 순서가 자동으로 보장된다. 이는 구현도 간단하고 성능도 우수하다. 다음 챕터에서 자세히 다룬다.

**글로벌 상태 접근 문제**

여러 로직 스레드가 공유 데이터(예: 글로벌 아이템 경매장, 랭킹 시스템)에 접근하면 동기화가 필요하다. 이 경우 다음 전략을 사용한다.

- **분할(Sharding)**: 데이터를 여러 샤드로 나누고 각 샤드를 하나의 스레드가 관리한다. 예를 들어 아이템 ID의 해시값으로 샤드를 결정한다.

**JobQueue.h**

```cpp
#pragma once
#include <concurrent_queue.h>
#include <functional>
#include <atomic>
#include <semaphore>

struct Job {
    std::function<void()> task;
    uint64_t sessionId;
    uint64_t sequenceNumber;
    
    Job() : sessionId(0), sequenceNumber(0) {}
    
    Job(std::function<void()> t, uint64_t sid, uint64_t seq)
        : task(std::move(t)), sessionId(sid), sequenceNumber(seq) {}
};

class JobQueue {
private:
    concurrency::concurrent_queue<Job> queue_;
    std::counting_semaphore<> semaphore_{0};
    std::atomic<bool> isShutdown_;
    std::atomic<uint64_t> totalEnqueued_;
    std::atomic<uint64_t> totalDequeued_;

public:
    JobQueue() : isShutdown_(false), totalEnqueued_(0), totalDequeued_(0) {}
    
    void Enqueue(Job job) {
        if (isShutdown_.load(std::memory_order_acquire)) {
            return;
        }
        
        queue_.push(std::move(job));
        totalEnqueued_.fetch_add(1, std::memory_order_relaxed);
        semaphore_.release();
    }
    
    bool Dequeue(Job& outJob) {
        semaphore_.acquire();
        
        if (isShutdown_.load(std::memory_order_acquire)) {
            return false;
        }
        
        bool success = queue_.try_pop(outJob);
        if (success) {
            totalDequeued_.fetch_add(1, std::memory_order_relaxed);
        }
        
        return success;
    }
    
    size_t GetPendingCount() const {
        return totalEnqueued_.load(std::memory_order_relaxed) - 
               totalDequeued_.load(std::memory_order_relaxed);
    }
    
    void Shutdown() {
        isShutdown_.store(true, std::memory_order_release);
        
        // 모든 대기 중인 스레드 깨우기
        for (int i = 0; i < 100; ++i) {
            semaphore_.release();
        }
    }
};
```

        thread_ = std::thread(&LogicWorker::WorkerThreadFunc, this);

        isRunning_ = false;
        if (thread_.joinable()) {
            thread_.join();
        }
    }

        return processedJobCount_.load(std::memory_order_relaxed);
    }
    
    double GetAverageProcessingTimeMs() const {
        uint64_t count = processedJobCount_.load(std::memory_order_relaxed);
        if (count == 0) return 0.0;
        return static_cast<double>(totalProcessingTimeUs_.load(std::memory_order_relaxed)) 
               / (count * 1000.0);
    }

private:

#include "LogicWorker.h"
#include <iostream>
#include <chrono>

void LogicWorker::WorkerThreadFunc() {
    std::cout << "[LogicWorker " << workerId_ << "] Started\n";
    
    while (isRunning_) {
        Job job;
        
        if (!pJobQueue_->Dequeue(job)) {
            // 종료 신호
            break;
        }
        
        auto startTime = std::chrono::high_resolution_clock::now();
        
        try {
            if (job.task) {
                job.task();
            }
        }
        catch (const std::exception& e) {
            std::cerr << "[LogicWorker " << workerId_ 
                      << "] Exception: " << e.what() << "\n";
        }
        catch (...) {
            std::cerr << "[LogicWorker " << workerId_ 
                      << "] Unknown exception\n";
        }
        
        auto endTime = std::chrono::high_resolution_clock::now();
        auto elapsedUs = std::chrono::duration_cast<std::chrono::microseconds>(
            endTime - startTime).count();
        
        processedJobCount_.fetch_add(1, std::memory_order_relaxed);
        totalProcessingTimeUs_.fetch_add(elapsedUs, std::memory_order_relaxed);
        
        // 너무 오래 걸린 작업 경고 (10ms 이상)
        if (elapsedUs > 10000) {
            std::cout << "[LogicWorker " << workerId_ 
                      << "] Long job detected: " 
                      << (elapsedUs / 1000.0) << "ms (SessionId: " 
                      << job.sessionId << ")\n";
        }
    }
    
    std::cout << "[LogicWorker " << workerId_ << "] Exiting. "
              << "Processed " << processedJobCount_ << " jobs.\n";
}
