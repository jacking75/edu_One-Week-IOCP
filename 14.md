# Chapter 14. 최종 테스트와 성능 측정
일주일간의 여정을 통해 IOCP 기반 오목 게임 서버를 완성했다. 이제 마지막 단계로 서버의 성능을 측정하고, 병목 지점을 찾아 최적화하는 방법을 학습한다. 성능 측정은 단순히 숫자를 확인하는 것이 아니라, 서버가 실제 운영 환경에서 얼마나 많은 사용자를 감당할 수 있는지, 어떤 부분을 개선해야 하는지를 파악하는 중요한 과정이다.

게임 서버는 실시간 응답성이 생명이다. 사용자가 돌을 놓았을 때 100ms 이내에 응답이 오지 않으면 게임 경험이 크게 저하된다. 따라서 처리량(Throughput)뿐만 아니라 지연시간(Latency)도 중요한 지표다. 이 장에서는 더미 클라이언트를 구현하여 수백, 수천 명의 동시 접속자를 시뮬레이션하고, Visual Studio의 프로파일링 도구를 활용하여 CPU, 메모리, 스레드 경합 등을 분석한다.

## 14.1 성능 측정 지표 정의 (TPS, Latency, CPU Usage)
서버 성능을 객관적으로 평가하기 위해서는 명확한 지표가 필요하다. 게임 서버에서 일반적으로 사용하는 주요 성능 지표는 다음과 같다.

**TPS (Transactions Per Second)**

TPS는 초당 처리할 수 있는 트랜잭션 수를 의미한다. 게임 서버에서는 초당 처리하는 패킷 수, 게임 액션 수 등으로 해석할 수 있다. 오목 게임의 경우, 돌 놓기 액션의 TPS가 주요 지표가 된다. 예를 들어 1000 TPS는 초당 1000개의 돌 놓기 요청을 처리할 수 있다는 의미다.

TPS는 서버의 전체적인 처리 능력을 나타내는 지표로, 동시 접속자 수가 증가할수록 선형적으로 증가하는지, 특정 지점에서 포화되는지를 확인할 수 있다. IOCP 서버는 이론적으로 스레드 수에 비례하여 TPS가 증가하지만, 실제로는 락 경합, 메모리 할당, 네트워크 대역폭 등의 제약으로 인해 한계에 도달한다.

**Latency (지연시간)**

Latency는 클라이언트가 요청을 보낸 시점부터 응답을 받는 시점까지의 시간이다. 게임 서버에서는 평균 지연시간보다 99 백분위수(99th percentile) 지연시간이 더 중요하다. 대부분의 요청이 10ms 이내에 처리되더라도, 1%의 요청이 1초 이상 걸린다면 사용자 경험이 크게 저하되기 때문이다.

지연시간을 측정할 때는 클라이언트 측에서 요청 전송 시각을 기록하고, 응답 수신 시각을 비교한다. 네트워크 RTT(Round Trip Time)를 제외한 순수 서버 처리 시간을 측정하려면, 서버 측에서 패킷 수신 시각과 응답 전송 시각을 로깅해야 한다.

```cpp
struct LatencyMetric {
    std::chrono::steady_clock::time_point requestTime;
    std::chrono::steady_clock::time_point responseTime;
    
    int64_t GetLatencyMs() const {
        return std::chrono::duration_cast<std::chrono::milliseconds>(
            responseTime - requestTime).count();
    }
};
```

**CPU Usage (CPU 사용률)**

CPU 사용률은 서버가 시스템 자원을 얼마나 효율적으로 사용하는지 나타낸다. IOCP 서버는 CPU 코어 수만큼 워커 스레드를 생성하므로, 이론적으로 100% CPU 사용률에 도달할 수 있다. 하지만 실제로는 락 대기, I/O 대기 등으로 인해 낮은 사용률을 보일 수 있다.

CPU 사용률을 측정하는 방법은 여러 가지가 있다. Windows에서는 `GetProcessTimes` API를 사용하여 프로세스의 CPU 시간을 측정하거나, 성능 카운터(Performance Counter)를 사용할 수 있다. Visual Studio Profiler는 자동으로 CPU 사용률을 측정하고 시각화한다.

**Memory Usage (메모리 사용량)**

메모리 사용량은 서버가 사용하는 물리 메모리와 가상 메모리를 의미한다. 메모리 누수(Memory Leak)가 있으면 시간이 지남에 따라 메모리 사용량이 계속 증가하여 결국 서버가 다운된다. 따라서 장시간 부하 테스트를 수행하면서 메모리 사용량이 안정적인지 확인해야 한다.

Windows에서는 `GetProcessMemoryInfo` API를 사용하여 프로세스의 메모리 사용량을 조회할 수 있다. Working Set, Private Bytes, Virtual Size 등의 지표를 확인한다.

**Connection Count (동시 접속자 수)**

게임 서버가 동시에 몇 명의 사용자를 수용할 수 있는지를 나타낸다. IOCP는 수만 개의 동시 연결을 처리할 수 있지만, 게임 로직의 복잡도, 메모리 제약 등으로 인해 실제 수용 가능한 인원은 제한된다.

**성능 측정 클래스 구현**

서버에 성능 측정 기능을 내장하여 실시간으로 지표를 수집한다.

```cpp
class PerformanceMonitor {
public:
    void RecordPacketProcessed() {
        packetsProcessed_.fetch_add(1, std::memory_order_relaxed);
    }

    void RecordLatency(int64_t latencyMs) {
        std::lock_guard<std::mutex> lock(latencyMutex_);
        latencies_.push_back(latencyMs);
    }

    void PrintMetrics() {
        auto now = std::chrono::steady_clock::now();
        auto elapsed = std::chrono::duration_cast<std::chrono::seconds>(
            now - startTime_).count();
        
        if (elapsed == 0) return;

        int64_t packets = packetsProcessed_.load();
        double tps = static_cast<double>(packets) / elapsed;

        std::lock_guard<std::mutex> lock(latencyMutex_);
        if (latencies_.empty()) return;

        std::sort(latencies_.begin(), latencies_.end());
        int64_t avgLatency = std::accumulate(latencies_.begin(), 
            latencies_.end(), 0LL) / latencies_.size();
        int64_t p50Latency = latencies_[latencies_.size() / 2];
        int64_t p99Latency = latencies_[latencies_.size() * 99 / 100];

        std::cout << "=== Performance Metrics ===" << std::endl;
        std::cout << "TPS: " << tps << std::endl;
        std::cout << "Avg Latency: " << avgLatency << "ms" << std::endl;
        std::cout << "P50 Latency: " << p50Latency << "ms" << std::endl;
        std::cout << "P99 Latency: " << p99Latency << "ms" << std::endl;
    }

private:
    std::atomic<int64_t> packetsProcessed_{0};
    std::vector<int64_t> latencies_;
    std::mutex latencyMutex_;
    std::chrono::steady_clock::time_point startTime_ = 
        std::chrono::steady_clock::now();
};
```

이 클래스를 서버에 통합하여 주기적으로 메트릭을 출력하거나, 외부 모니터링 시스템으로 전송할 수 있다.

## 14.2 더미 클라이언트 구현
실제 사용자를 시뮬레이션하는 더미 클라이언트를 구현하여 부하 테스트를 수행한다. 더미 클라이언트는 서버에 연결하고, 패킷을 전송하며, 응답을 받아 지연시간을 측정한다. 수백, 수천 개의 더미 클라이언트를 동시에 실행하여 서버의 한계를 테스트한다.

**더미 클라이언트 아키텍처**

더미 클라이언트는 실제 게임 로직을 구현하지 않고, 서버에 요청을 보내고 응답을 받는 최소한의 기능만 구현한다. 각 클라이언트는 독립적인 소켓을 사용하며, 비동기 I/O 또는 멀티스레드 방식으로 동시에 여러 클라이언트를 실행한다.

```cpp
class DummyClient {
public:
    DummyClient(int id) : id_(id) {}

    bool Connect(const char* ip, int port) {
        socket_ = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP);
        if (socket_ == INVALID_SOCKET) return false;

        sockaddr_in addr = {};
        addr.sin_family = AF_INET;
        addr.sin_port = htons(static_cast<u_short>(port));
        inet_pton(AF_INET, ip, &addr.sin_addr);

        if (connect(socket_, (sockaddr*)&addr, sizeof(addr)) == SOCKET_ERROR) {
            closesocket(socket_);
            return false;
        }

        return true;
    }

    void Run() {
        std::thread sendThread([this]() { SendLoop(); });
        std::thread recvThread([this]() { RecvLoop(); });

        sendThread.join();
        recvThread.join();
    }

private:
    void SendLoop() {
        while (running_) {
            // 랜덤한 위치에 돌 놓기 패킷 전송
            int row = rand() % 19;
            int col = rand() % 19;

            PlaceStonePacket packet;
            packet.header.type = PacketType::PlaceStone;
            packet.header.size = sizeof(PlaceStonePacket);
            packet.row = row;
            packet.col = col;

            auto sendTime = std::chrono::steady_clock::now();
            
            send(socket_, (char*)&packet, sizeof(packet), 0);

            {
                std::lock_guard<std::mutex> lock(pendingMutex_);
                pendingRequests_[requestId_++] = sendTime;
            }

            std::this_thread::sleep_for(std::chrono::milliseconds(100));
        }
    }

    void RecvLoop() {
        std::vector<char> buffer;
        buffer.reserve(4096);

        char tempBuffer[4096];
        while (running_) {
            int bytesReceived = recv(socket_, tempBuffer, sizeof(tempBuffer), 0);
            if (bytesReceived <= 0) break;

            buffer.insert(buffer.end(), tempBuffer, tempBuffer + bytesReceived);

            while (buffer.size() >= sizeof(PacketHeader)) {
                PacketHeader* header = (PacketHeader*)buffer.data();
                if (buffer.size() < header->size) break;

                ProcessPacket(buffer.data());

                buffer.erase(buffer.begin(), buffer.begin() + header->size);
            }
        }
    }

    void ProcessPacket(const char* data) {
        PacketHeader* header = (PacketHeader*)data;

        if (header->type == PacketType::PlaceStoneResult) {
            auto recvTime = std::chrono::steady_clock::now();

            std::lock_guard<std::mutex> lock(pendingMutex_);
            if (!pendingRequests_.empty()) {
                auto sendTime = pendingRequests_.begin()->second;
                auto latency = std::chrono::duration_cast<std::chrono::milliseconds>(
                    recvTime - sendTime).count();

                latencies_.push_back(latency);
                pendingRequests_.erase(pendingRequests_.begin());
            }
        }
    }

    int id_;
    SOCKET socket_ = INVALID_SOCKET;
    bool running_ = true;
    int requestId_ = 0;
    
    std::map<int, std::chrono::steady_clock::time_point> pendingRequests_;
    std::vector<int64_t> latencies_;
    std::mutex pendingMutex_;
};
```

**다중 클라이언트 관리**

여러 개의 더미 클라이언트를 생성하고 관리하는 매니저 클래스를 구현한다.

```cpp
class LoadTestManager {
public:
    void StartTest(const char* serverIP, int port, int clientCount) {
        std::cout << "Starting load test with " << clientCount << " clients..." 
                  << std::endl;

        for (int i = 0; i < clientCount; ++i) {
            auto client = std::make_unique<DummyClient>(i);
            
            if (!client->Connect(serverIP, port)) {
                std::cerr << "Client " << i << " failed to connect" << std::endl;
                continue;
            }

            std::thread clientThread([client = client.get()]() {
                client->Run();
            });

            clients_.push_back(std::move(client));
            threads_.push_back(std::move(clientThread));

            // 연결 간격을 두어 서버 부하 분산
            std::this_thread::sleep_for(std::chrono::milliseconds(10));
        }

        std::cout << clientCount << " clients connected successfully" << std::endl;
    }

    void WaitForCompletion() {
        for (auto& thread : threads_) {
            if (thread.joinable()) {
                thread.join();
            }
        }
    }

    void PrintStatistics() {
        std::vector<int64_t> allLatencies;
        
        for (auto& client : clients_) {
            auto latencies = client->GetLatencies();
            allLatencies.insert(allLatencies.end(), 
                latencies.begin(), latencies.end());
        }

        if (allLatencies.empty()) return;

        std::sort(allLatencies.begin(), allLatencies.end());

        int64_t total = std::accumulate(allLatencies.begin(), 
            allLatencies.end(), 0LL);
        double avg = static_cast<double>(total) / allLatencies.size();
        int64_t p50 = allLatencies[allLatencies.size() / 2];
        int64_t p95 = allLatencies[allLatencies.size() * 95 / 100];
        int64_t p99 = allLatencies[allLatencies.size() * 99 / 100];

        std::cout << "\n=== Load Test Results ===" << std::endl;
        std::cout << "Total Requests: " << allLatencies.size() << std::endl;
        std::cout << "Average Latency: " << avg << "ms" << std::endl;
        std::cout << "P50 Latency: " << p50 << "ms" << std::endl;
        std::cout << "P95 Latency: " << p95 << "ms" << std::endl;
        std::cout << "P99 Latency: " << p99 << "ms" << std::endl;
    }

private:
    std::vector<std::unique_ptr<DummyClient>> clients_;
    std::vector<std::thread> threads_;
};
```

**더미 클라이언트 실행**

```cpp
int main() {
    WSADATA wsaData;
    WSAStartup(MAKEWORD(2, 2), &wsaData);

    LoadTestManager manager;
    manager.StartTest("127.0.0.1", 9000, 100);  // 100개 클라이언트

    // 60초간 테스트 실행
    std::this_thread::sleep_for(std::chrono::seconds(60));

    manager.WaitForCompletion();
    manager.PrintStatistics();

    WSACleanup();
    return 0;
}
```

이렇게 구현한 더미 클라이언트로 서버의 동시 접속 처리 능력, 평균 응답 시간, 백분위수 지연시간 등을 측정할 수 있다.

## 14.3 부하 테스트 시나리오
부하 테스트는 단순히 많은 클라이언트를 연결하는 것이 아니라, 실제 게임 플레이 패턴을 시뮬레이션해야 의미 있는 결과를 얻을 수 있다. 오목 게임의 경우 다음과 같은 시나리오를 구성한다.

**시나리오 1: 점진적 부하 증가 (Ramp-Up Test)**

처음에는 소수의 클라이언트로 시작하여 점진적으로 연결 수를 늘려간다. 서버가 어느 시점부터 성능 저하를 보이는지, CPU 사용률이 포화되는지를 확인한다.

```cpp
void RampUpTest(const char* serverIP, int port) {
    LoadTestManager manager;
    
    for (int i = 100; i <= 1000; i += 100) {
        std::cout << "\n=== Testing with " << i << " clients ===" << std::endl;
        
        manager.StartTest(serverIP, port, i);
        std::this_thread::sleep_for(std::chrono::seconds(30));
        
        manager.PrintStatistics();
        manager.StopAll();
        
        std::this_thread::sleep_for(std::chrono::seconds(5));
    }
}
```

**시나리오 2: 스파이크 테스트 (Spike Test)**

갑작스럽게 많은 사용자가 동시에 접속하는 상황을 시뮬레이션한다. 이벤트 시작, 게임 출시 등의 상황에서 발생할 수 있다.

```cpp
void SpikeTest(const char* serverIP, int port) {
    LoadTestManager manager;
    
    // 기본 부하 (100 클라이언트)
    manager.StartTest(serverIP, port, 100);
    std::this_thread::sleep_for(std::chrono::seconds(10));
    
    std::cout << "Triggering spike..." << std::endl;
    
    // 갑작스러운 부하 증가 (추가 500 클라이언트)
    manager.StartTest(serverIP, port, 500);
    std::this_thread::sleep_for(std::chrono::seconds(30));
    
    manager.PrintStatistics();
}
```

**시나리오 3: 장시간 안정성 테스트 (Soak Test)**

일정한 부하를 장시간(수 시간~수 일) 유지하면서 메모리 누수, 리소스 고갈 등의 문제가 없는지 확인한다.

```cpp
void SoakTest(const char* serverIP, int port, int hours) {
    LoadTestManager manager;
    manager.StartTest(serverIP, port, 200);
    
    auto endTime = std::chrono::steady_clock::now() + 
                   std::chrono::hours(hours);
    
    while (std::chrono::steady_clock::now() < endTime) {
        std::this_thread::sleep_for(std::chrono::minutes(10));
        
        // 주기적으로 메트릭 출력
        manager.PrintStatistics();
        
        // 메모리 사용량 확인
        PROCESS_MEMORY_COUNTERS pmc;
        GetProcessMemoryInfo(GetCurrentProcess(), &pmc, sizeof(pmc));
        std::cout << "Memory Usage: " << pmc.WorkingSetSize / (1024 * 1024) 
                  << " MB" << std::endl;
    }
}
```

**시나리오 4: 실제 게임 플레이 시뮬레이션**

더미 클라이언트가 단순히 랜덤한 패킷을 보내는 것이 아니라, 실제 게임 플레이 패턴을 따르도록 구현한다. 매칭 대기, 게임 시작, 턴제 플레이, 게임 종료 등의 시퀀스를 시뮬레이션한다.

```cpp
class RealisticDummyClient : public DummyClient {
protected:
    void GamePlayLoop() override {
        // 1. 매칭 대기
        WaitForGameStart();
        
        // 2. 게임 진행 (평균 50턴)
        for (int turn = 0; turn < 50; ++turn) {
            if (IsMyTurn()) {
                // 사고 시간 (1~3초)
                std::this_thread::sleep_for(
                    std::chrono::seconds(1 + rand() % 3));
                
                PlaceRandomStone();
            }
            
            WaitForOpponentMove();
            
            if (IsGameOver()) break;
        }
        
        // 3. 대기 시간 후 다시 매칭
        std::this_thread::sleep_for(std::chrono::seconds(5));
    }
};
```

이러한 시나리오를 통해 서버의 다양한 측면을 테스트하고, 실제 운영 환경에서 발생할 수 있는 문제를 미리 발견할 수 있다.

## 14.4 Visual Studio Profiler 활용
Visual Studio 2022는 강력한 프로파일링 도구를 제공하여 CPU 사용률, 메모리 할당, 스레드 경합 등을 시각화하고 분석할 수 있다. 프로파일링은 코드의 어느 부분이 가장 많은 시간을 소비하는지(핫스팟), 어떤 락이 병목을 일으키는지를 정확히 파악할 수 있게 해준다.

**CPU 프로파일링**

CPU 프로파일러는 각 함수가 실행에 소요한 시간을 측정하여 성능 병목을 찾는다. Visual Studio에서 프로파일링을 시작하는 방법은 다음과 같다.

1. 메뉴에서 **Debug > Performance Profiler** 선택
2. **CPU Usage** 체크박스 선택
3. **Start** 버튼 클릭
4. 서버를 실행하고 부하 테스트 수행
5. 충분한 샘플을 수집한 후 **Stop Collection**

프로파일링 결과는 함수 호출 트리와 CPU 시간 비율로 표시된다. **Hot Path** 기능은 가장 많은 CPU 시간을 소비하는 호출 경로를 강조 표시한다.

```
Top CPU Consuming Functions:
┌────────────────────────────────────┬─────────┬──────────┐
│ Function                           │ % Time  │ Samples  │
├────────────────────────────────────┼─────────┼──────────┤
│ ProcessPacket                      │  35.2%  │  24,531  │
│ SendPacket                         │  18.7%  │  13,042  │
│ CheckWinCondition                  │  12.4%  │   8,652  │
│ IOCPWorkerThread                   │   8.9%  │   6,204  │
│ std::mutex::lock                   │   7.3%  │   5,091  │
└────────────────────────────────────┴─────────┴──────────┘
```

만약 `std::mutex::lock`이 상위에 랭크되어 있다면, 락 경합이 심각하다는 신호다. 이 경우 락 범위를 줄이거나, Lock-Free 자료구조를 사용하거나, 락 분할(Lock Striping) 기법을 적용해야 한다.

**메모리 프로파일링**

메모리 프로파일러는 할당/해제 패턴을 분석하여 메모리 누수와 과도한 할당을 찾는다.

1. **Memory Usage** 프로파일러 선택
2. 프로파일링 시작
3. 부하 테스트 수행
4. 스냅샷을 여러 번 촬영하여 메모리 증가 추이 확인

프로파일링 결과에서 다음을 확인한다:

- **Heap Size**: 시간에 따른 힙 크기 변화. 지속적으로 증가하면 메모리 누수 의심
- **Allocation Count**: 할당 횟수가 과도하면 성능 저하 원인
- **Object Lifetime**: 객체가 예상보다 오래 살아있으면 리소스 정리 문제

```cpp
// 메모리 누수 예시 - Session 객체가 해제되지 않음
void BadExample() {
    Session* session = new Session();
    sessions_.push_back(session);
    // delete를 호출하지 않아 메모리 누수 발생
}

// 올바른 예시 - 스마트 포인터 사용
void GoodExample() {
    auto session = std::make_shared<Session>();
    sessions_.push_back(session);
    // shared_ptr이 자동으로 메모리 관리
}
```

**동시성 프로파일링**

동시성 프로파일러는 스레드 간 상호작용, 락 경합, 데드락 등을 시각화한다.

1. **Concurrency Visualizer** 프로파일러 선택 (별도 설치 필요할 수 있음)
2. 프로파일링 시작
3. 부하 테스트 수행

결과에서 다음을 확인한다:

- **Thread Timeline**: 각 스레드가 실행/대기/블로킹 상태인 시간대
- **Lock Contention**: 어떤 락에서 경합이 발생하는지
- **Thread Synchronization**: 스레드 간 동기화 패턴

```
Thread Timeline (IOCP Workers):
Thread 1: ████████░░░░░░░░████████░░░░░░░░
Thread 2: ░░░░░░░░████████░░░░░░░░████████
Thread 3: ░░░░████████░░░░░░░░████████░░░░
Thread 4: ████████░░░░████████░░░░░░░░████

█ = Executing
░ = Waiting on Lock
```

만약 워커 스레드들이 대부분의 시간을 락 대기에 소비한다면, 락 설계를 재검토해야 한다.

**프로파일링 모범 사례**

프로파일링을 효과적으로 수행하기 위한 팁은 다음과 같다:

- **Release 빌드로 프로파일링**: Debug 빌드는 최적화가 되어 있지 않아 부정확한 결과를 낳는다. Release 빌드에 디버그 심볼(PDB)을 포함시켜 함수 이름을 확인할 수 있도록 한다.

- **충분한 샘플 수집**: 너무 짧게 프로파일링하면 통계적으로 의미 없는 결과가 나온다. 최소 30초~1분 이상 프로파일링한다.

- **반복 측정**: 한 번의 프로파일링 결과로 결론을 내리지 않고, 여러 번 측정하여 일관성을 확인한다.

- **단계적 최적화**: 가장 큰 병목부터 해결하고, 다시 프로파일링하여 효과를 확인한다. 작은 최적화에 시간을 낭비하지 않는다.

## 14.5 병목 지점 분석과 최적화

프로파일링 결과를 바탕으로 병목 지점을 찾았다면, 이를 최적화하는 단계다. IOCP 게임 서버에서 흔히 발견되는 병목과 해결 방법을 살펴본다.

**병목 1: 락 경합 (Lock Contention)**

여러 스레드가 동일한 뮤텍스를 획득하려고 경쟁하면 성능이 크게 저하된다. 특히 IOCP 워커 스레드가 많을수록 락 경합은 심각해진다.

**증상**: 프로파일러에서 `std::mutex::lock` 또는 `EnterCriticalSection`이 상위 랭크. 동시성 프로파일러에서 스레드들이 대부분 락 대기 상태.

**해결 방법**:

- **락 범위 최소화**: 크리티컬 섹션 내부에서 불필요한 작업을 제거한다.

```cpp
// Bad: 락을 오래 잡고 있음
void BadSendPacket(const Packet& packet) {
    std::lock_guard<std::mutex> lock(sendMutex_);
    SerializePacket(packet);  // 시간이 오래 걸리는 작업
    send(socket_, buffer, size, 0);
}

// Good: 락 범위 최소화
void GoodSendPacket(const Packet& packet) {
    char buffer[1024];
    int size = SerializePacket(packet, buffer);  // 락 밖에서 실행
    
    std::lock_guard<std::mutex> lock(sendMutex_);
    send(socket_, buffer, size, 0);
}
```

- **락 분할 (Lock Striping)**: 하나의 큰 락 대신 여러 개의 작은 락으로 분할한다.

```cpp
// 세션을 ID 범위별로 분할하여 락 경합 감소
class SessionManager {
    static constexpr int LOCK_COUNT = 16;
    std::mutex mutexes_[LOCK_COUNT];
    std::unordered_map<int, Session*> sessions_[LOCK_COUNT];
    
public:
    void AddSession(int id, Session* session) {
        int index = id % LOCK_COUNT;
        std::lock_guard<std::mutex> lock(mutexes_[index]);
        sessions_[index][id] = session;
    }
};
```

- **Lock-Free 자료구조 사용**: Concurrency Runtime의 `concurrent_queue`, `concurrent_unordered_map` 등을 사용한다.

```cpp
#include <concurrent_queue.h>

// Lock-Free 큐 사용
concurrency::concurrent_queue<Packet*> packetQueue_;

void ProducerThread() {
    Packet* packet = new Packet();
    packetQueue_.push(packet);  // 락 없이 안전
}

void ConsumerThread() {
    Packet* packet;
    if (packetQueue_.try_pop(packet)) {
        ProcessPacket(packet);
    }
}
```

**병목 2: 메모리 할당 (Memory Allocation)**

빈번한 메모리 할당/해제는 성능 저하의 주요 원인이다. 특히 멀티스레드 환경에서 malloc/free는 내부적으로 락을 사용하므로 경합이 발생한다.

**증상**: 프로파일러에서 `malloc`, `free`, `operator new`, `operator delete`가 상위 랭크. 메모리 프로파일러에서 할당 횟수가 매우 높음.

**해결 방법**:

- **객체 풀 (Object Pool)**: 미리 객체를 생성해두고 재사용한다.

```cpp
template<typename T>
class ObjectPool {
public:
    ObjectPool(size_t initialSize) {
        for (size_t i = 0; i < initialSize; ++i) {
            pool_.push(new T());
        }
    }

    T* Acquire() {
        T* obj;
        if (pool_.try_pop(obj)) {
            return obj;
        }
        return new T();  // 풀이 비었으면 새로 생성
    }

    void Release(T* obj) {
        obj->Reset();  // 객체 상태 초기화
        pool_.push(obj);
    }

private:
    concurrency::concurrent_queue<T*> pool_;
};

// 사용 예시
ObjectPool<Packet> packetPool(1000);

void SendPacket() {
    Packet* packet = packetPool.Acquire();
    // 패킷 사용
    packetPool.Release(packet);
}
```

- **메모리 풀 (Memory Pool)**: 고정 크기 블록을 미리 할당하여 관리한다.

- **스택 할당 활용**: 가능하면 힙 대신 스택에 할당한다.

```cpp
// Bad: 힙 할당
void ProcessPacket() {
    char* buffer = new char[1024];
    // 작업
    delete[] buffer;
}

// Good: 스택 할당
void ProcessPacket() {
    char buffer[1024];
    // 작업
}
```

**병목 3: 불필요한 복사 (Excessive Copying)**

대용량 데이터를 복사하면 CPU와 메모리 대역폭을 낭비한다.

**해결 방법**:

- **이동 의미론 사용**: `std::move`를 활용하여 복사 대신 이동한다.

```cpp
// Bad: 복사 발생
std::vector<char> GetPacketData() {
    std::vector<char> data(1024);
    // 데이터 채우기
    return data;  // 복사 발생 가능
}

// Good: 이동 발생
std::vector<char> GetPacketData() {
    std::vector<char> data(1024);
    // 데이터 채우기
    return data;  // RVO 또는 이동 발생
}

void UseData() {
    auto data = GetPacketData();  // 복사 없이 이동
}
```

- **참조 전달**: 함수 매개변수로 큰 객체를 전달할 때 참조나 포인터를 사용한다.

```cpp
// Bad: 복사 전달
void ProcessPacket(Packet packet) {
    // packet이 복사됨
}

// Good: const 참조 전달
void ProcessPacket(const Packet& packet) {
    // 복사 없음
}
```

**병목 4: 컨텍스트 스위칭 (Context Switching)**

과도한 스레드 수는 컨텍스트 스위칭 오버헤드를 증가시킨다.

**해결 방법**:

- **적절한 스레드 수**: IOCP 워커 스레드는 CPU 코어 수의 1~2배가 적당하다.

```cpp
SYSTEM_INFO sysInfo;
GetSystemInfo(&sysInfo);
int workerThreadCount = sysInfo.dwNumberOfProcessors * 2;
```

- **스레드 풀 사용**: 작업마다 스레드를 생성하지 않고 스레드 풀을 재사용한다.

**최적화 후 검증**

최적화를 적용한 후 반드시 다시 프로파일링하여 효과를 측정한다. 때로는 최적화가 예상과 다른 결과를 낳거나, 다른 부분에서 새로운 병목이 발생할 수 있다.

```
Before Optimization:
- TPS: 5,000
- P99 Latency: 150ms
- CPU Usage: 60%

After Lock-Free Queue:
- TPS: 12,000 (140% 증가)
- P99 Latency: 45ms (70% 감소)
- CPU Usage: 85%
```

## 14.6 향후 학습 방향
일주일간의 IOCP 게임 서버 프로그래밍 여정을 마쳤다. 기본적인 IOCP 서버 구현부터 오목 게임 서버 완성까지, 실무에 필요한 핵심 개념과 기법을 학습했다. 하지만 게임 서버 개발의 세계는 여기서 끝나지 않는다. 더 깊이 있는 학습을 위한 방향을 제시한다.

**1. 분산 서버 아키텍처**

단일 서버로는 수만 명의 동시 접속자를 처리하기 어렵다. 로드 밸런서, 게임 서버 클러스터, 데이터베이스 샤딩 등을 학습하여 수평 확장 가능한 아키텍처를 구축한다.

**학습 주제**:
- 로드 밸런싱 (Round-Robin, Least Connection, Consistent Hashing)
- 서버 간 통신 (RPC, Message Queue)
- 세션 관리 (Redis, Memcached)
- 데이터베이스 복제와 샤딩

**2. 고급 네트워크 최적화**

게임 장르에 따라 다양한 네트워크 최적화 기법이 필요하다. FPS 게임은 초저지연, MMORPG는 대규모 동시 접속, 실시간 전략 게임은 동기화 등 각기 다른 요구사항을 갖는다.

**학습 주제**:
- UDP 기반 신뢰성 프로토콜 (KCP, ENet)
- 클라이언트 예측과 서버 조정 (Client-Side Prediction, Server Reconciliation)
- 상태 동기화 기법 (Delta Compression, Interest Management)
- Anti-Cheat와 보안

**3. 데이터베이스 연동**

게임 서버는 사용자 정보, 게임 기록, 랭킹 등을 영구 저장해야 한다. 비동기 데이터베이스 I/O, 트랜잭션 처리, 캐싱 전략 등을 학습한다.

**학습 주제**:
- 비동기 데이터베이스 쿼리 (ODBC, MySQL Connector)
- ORM과 쿼리 빌더
- 데이터베이스 연결 풀
- NoSQL 활용 (MongoDB, Redis)

**4. 게임 로직 최적화**

복잡한 게임 로직을 효율적으로 구현하는 기법을 학습한다. 공간 분할, 이벤트 시스템, 엔티티 컴포넌트 시스템 등이 있다.

**학습 주제**:
- Spatial Partitioning (Quadtree, Octree, Grid)
- Entity Component System (ECS)
- 이벤트 드리븐 아키텍처
- 게임 물리 최적화

**5. 모니터링과 운영**

서비스 중인 게임 서버를 안정적으로 운영하기 위한 모니터링, 로깅, 장애 대응 체계를 구축한다.

**학습 주제**:
- 로그 수집과 분석 (ELK Stack, Splunk)
- 메트릭 모니터링 (Prometheus, Grafana)
- 알람과 장애 대응
- 무중단 배포 (Blue-Green, Rolling Update)

**6. 다른 플랫폼과 기술**

Windows와 IOCP 외에도 다양한 플랫폼과 기술을 학습하여 시야를 넓힌다.

**학습 주제**:
- Linux와 epoll
- 크로스 플랫폼 네트워크 라이브러리 (Boost.Asio, libuv)
- 게임 엔진 통합 (Unreal Engine, Unity)
- 클라우드 인프라 (AWS, Azure, GCP)

**추천 학습 자료**

- **오픈소스 프로젝트**: libevent, Photon Server, Colyseus

**마무리**

IOCP는 Windows에서 고성능 네트워크 서버를 구현하는 강력한 도구다. 이 책에서 학습한 개념과 기법은 실무에서 바로 활용할 수 있는 실전 지식이다. 하지만 기술은 계속 발전하고, 게임 서버 개발은 끊임없이 새로운 도전을 제시한다. 지속적인 학습과 실습을 통해 더 나은 게임 서버 개발자로 성장하기를 바란다.

게임 서버 개발은 네트워크, 동시성, 알고리즘, 시스템 프로그래밍 등 컴퓨터 과학의 다양한 영역이 종합적으로 요구되는 도전적인 분야다. 이 책을 통해 습득한 기초 위에 지속적으로 학습하고 실험하면서, 수백만 명의 사용자가 즐길 수 있는 안정적이고 확장 가능한 게임 서버를 구축하는 역량을 키워나가기를 기대한다.

**Windows 성능 카운터**

```cpp
#include <pdh.h>
#pragma comment(lib, "pdh.lib")

class PerformanceCounter {
public:
    PerformanceCounter() {
        PdhOpenQuery(nullptr, 0, &query_);
        PdhAddCounter(query_, L"\\Processor(_Total)\\% Processor Time", 
                      0, &cpuCounter_);
        PdhCollectQueryData(query_);
    }

    double GetCPUUsage() {
        PDH_FMT_COUNTERVALUE value;
        PdhCollectQueryData(query_);
        PdhGetFormattedCounterValue(cpuCounter_, PDH_FMT_DOUBLE, 
                                     nullptr, &value);
        return value.doubleValue;
    }

private:
    PDH_HQUERY query_;
    PDH_HCOUNTER cpuCounter_;
};
```

**타이머 유틸리티**

```cpp
class ScopedTimer {
public:
    ScopedTimer(const std::string& name) 
        : name_(name), start_(std::chrono::high_resolution_clock::now()) {}

    ~ScopedTimer() {
        auto end = std::chrono::high_resolution_clock::now();
        auto duration = std::chrono::duration_cast<std::chrono::microseconds>(
            end - start_).count();
        std::cout << name_ << " took " << duration << " µs\n";
    }

private:
    std::string name_;
    std::chrono::high_resolution_clock::time_point start_;
};

// 사용 예시
void ExpensiveFunction() {
    ScopedTimer timer("ExpensiveFunction");

}
```
